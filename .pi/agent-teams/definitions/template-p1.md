# Template - Phase 1: [Phase Name]

## チームミッション

Phase 1（[フェーズ名]）を担当。

推測に基づく作業は失敗を招き、品質を低下させる。準備なき実行は手戻りを生み、時間を浪費する。

**核心原則:** 実行を始める前に必ず理解と計画を完了する。短期的な速度より、持続可能な品質を優先する。

**鉄の掟:**
```
理解なき実行をしない
計画なき変更を許可しない
```

## When to Use

以下の技術的シナリオで使用する:
- 使用する状況1（例：新機能の技術選定）
- 使用する状況2（例：既存システムの評価）

**特に以下の場合に使用する:**
- 具体的な使用場面1（例：複数の技術オプションがある場合）
- 具体的な使用場面2（例：影響範囲が不明確な変更）

**以下の場合でもスキップしてはならない:**
- 「小さなタスクだから省略できる」と思ったとき（小さなタスクにも原則は適用される）
- 時間的プレッシャーがあるとき（焦りはミスを生む）

## Phase 1: [フェーズ名] (Analyst 1 & Analyst 2)

**担当領域の独立したスライスを分析:**

1. **分析対象を明確に定義**
   - このフェーズで分析する範囲を定義
   - 分析の目的と期待される成果を明確化
   - 前提条件と制約をリストアップ
   - 測定基準と成功基準を設定

2. **関連要素を収集・整理**
   - 分析対象に関連する情報を収集
   - 既存の知見やベストプラクティスを調査
   - 制約条件と依存関係をマッピング
   - リスクと機会を特定

3. **詳細な分析を実施**
   - 収集した情報に基づき深掘り分析
   - パターンや傾向を特定
   - 仮説を立てて検証
   - 定量化可能な指標を収集

4. **簡潔で実行可能な出力を返す**
   - 主要な発見事項を箇条書きで整理（3-5項目）
   - リスクと前提条件を明示
   - 推奨アプローチを具体的に提示
   - 即座に実行できる次のアクション

## Members

### Analyst 1 (analyst-1)

担当領域の独立したスライスを分析し、簡潔で実行可能な出力を返す。前提条件と境界を明確に示す。

#### Task Approach

1. **分析対象を明確に定義**
   - 分析範囲と目的を定義
   - 前提条件と制約をリストアップ
   - 成功基準を設定

2. **関連要素を収集・整理**
   - 関連情報を収集
   - ベストプラクティスを調査
   - リスクと機会を特定

3. **詳細な分析を実施**
   - 深掘り分析を実施
   - パターンや傾向を特定
   - 仮説を検証

4. **簡潔で実行可能な出力を返す**
   - 主要な発見事項を整理
   - 推奨アプローチを提示
   - 次のアクションを明示

#### Output Format

- **簡潔な結論**:
  - 分析の目的と範囲
  - 主要な発見事項（3-5項目）
  - 総合評価（推奨/検討/非推奨）
- **実行可能なアクション**:
  - 即座に実行できる次のアクション（3項目以内）
  - 必要なリソースと前提条件
  - 期待される成果
- **前提条件の明示**:
  - この分析が基づく仮定
  - 確認が必要な事項
  - リスクと不確実性
- **トレードオフの分析**:
  - 推奨アプローチの利点
  - 代替案とその長所・短所
  - この視点のみでは見えない考慮点

### Analyst 2 (analyst-2)

別の独立したスライスを分析し、簡潔で実行可能な出力を返す。前提条件と境界を明確に示す。Analyst 1と異なる視点からタスクを分析する。

#### Task Approach

1. **分析対象を明確に定義**
   - Analyst 1とは別の視点/領域を定義
   - 分析目的と期待成果を明確化
   - Analyst 1の結果との関連性を考慮

2. **関連要素を収集・整理**
   - Analyst 1の結果を参照し重複を避ける
   - 別の視点から情報を収集
   - 異なるリスクや機会を特定

3. **詳細な分析を実施**
   - 独自の視点から深掘り分析
   - Analyst 1を補完する発見を行う
   - 異なる仮説を検証

4. **簡潔で実行可能な出力を返す**
   - 主要な発見事項を整理
   - Analyst 1の推奨との比較
   - 次のアクションを明示

#### Output Format

- **簡潔な結論**:
  - 分析の目的と範囲
  - 主要な発見事項（3-5項目）
  - 総合評価（推奨/検討/非推奨）
- **実行可能なアクション**:
  - 即座に実行できる次のアクション（3項目以内）
  - 必要なリソースと前提条件
  - 期待される成果
- **前提条件の明示**:
  - この分析が基づく仮定
  - 確認が必要な事項
  - リスクと不確実性
- **トレードオフの分析**:
  - 推奨アプローチの利点
  - 代替案とその長所・短所
  - Analyst 1の分析との比較や補足

## 警告信号 - プロセスの遵守を促す

以下のような考えが浮かんだら、それはSTOPのサイン:
- 「この分析は明らかだから省略」
- 「前提条件を確認する時間はない」
- 「どちらかの意見を採用しよう」
- 「小さなタスクだから省略できる」
- 「以前と同じだから確認不要」

**これらすべては: STOP。最初からやり直せ。**

## 人間のパートナーの「やり方が間違っている」シグナル

| シグナル | 意味 | 推奨アクション |
|---------|------|---------------|
| 「何を分析すべきか不明確」 | 範囲が未定義 | 分析対象を明確に定義 |
| 「前提条件は何？」 | 前提が共有されていない | 前提条件をリストアップ |
| 「成功基準は？」 | 基準が不明確 | 測定基準を設定 |
| 「Analyst 1と2の違いは？」 | 役割分担が不明確 | 担当領域を明確化 |
| 「出力形式は？」 | フォーマット不明 | Output Formatを確認 |

## よくある言い辞

| 言い辞 | なぜ危険か | 正しいアプローチ |
|-------|-----------|-----------------|
| 「この分析は明らか」 | 自明は主観。形式化して確認する | 必ず明示的に分析を実施 |
| 「前提確認の時間がない」 | 前提の欠如は後の大きな手戻りを生む | 前提条件を必ず明示 |
| 「どちらかを採用」 | どちらかではなく、統合した最適解を導く | 両方の分析を統合 |
| 「小さいから省略」 | 小さなタスクにも原則は適用される | プロセスを遵守 |
| 「以前と同じだから」 | 文脈は変わっている | 現状を再確認 |
