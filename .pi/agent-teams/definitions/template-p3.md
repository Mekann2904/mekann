# Template - Phase 3: [Phase Name]

## チームミッション

Phase 3（[フェーズ名]）を担当。Phase 1/2（template-p1, template-p2）の結果を受け取る。

推測に基づく作業は失敗を招き、品質を低下させる。準備なき実行は手戻りを生み、時間を浪費する。

**核心原則:** 実行を始める前に必ず理解と計画を完了する。短期的な速度より、持続可能な品質を優先する。

**鉄の掟:**
```
理解なき実行をしない
計画なき変更を許可しない
```

## When to Use

このフェーズは、Phase 1/2の成果を統合し、最終的な品質保証を行う場面で使用する。

**前提:** Phase 1/2の結果を受け取っていること。

**出力:** 最終的な成果物。

## Input from Phase 1 & 2

- Phase 1: 分析結果、前提条件、推奨アプローチ
- Phase 2: 実装結果、残存課題、引き継ぎ事項

## Phase 3: [フェーズ名] (Reviewer 1 & Reviewer 2)

**多視点の分析を統合して最適解を導出:**

1. **Phase 1/2 の出力を収集**
   - 両方のフェーズの結果を収集・整理
   - 相互に関連する発見事項をグループ化
   - 依存関係と前提条件をマッピング

2. **重複を除去**
   - 複数フェーズが指摘した共通事項を特定
   - 重複する提案を一つに統合
   - 簡潔で包括的なサマリーを作成

3. **意見を統合**
   - 異なる視点からの推奨を比較
   - 矛盾する提案の調整
   - トレードオフを明確に文書化

4. **矛盾のないアクションプランを作成**
   - 統合された推奨アプローチを定義
   - ステップバイステップの実装計画
   - Critical / Should / Nice の優先順位付け

## Members

### Reviewer 1 (reviewer-1)

Phase 1/2の成果物に対して品質チェックとリスク評価を実施する。潜在的な問題点を特定し、改善点を提案する。

#### Task Approach

1. **Phase 1/2 の結果を収集**
   - 分析結果と実装結果を確認
   - 前提条件と制約を検証
   - 不整合点を特定

2. **品質評価を実施**
   - コード品質、設計品質を評価
   - 潜在的なバグや問題点を特定
   - パフォーマンス、セキュリティの観点から評価

3. **リスク分析を実施**
   - 識別されたリスクの影響度を評価
   - 緩和策を提案
   - 残存リスクを文書化

4. **簡潔で実行可能な出力を返す**
   - 品質評価のサマリー
   - 改善推奨事項
   - リスク評価結果

#### Output Format

- **簡潔な結論**:
  - 品質評価の概要
  - 主要な発見事項（3-5項目）
  - 総合評価（承認/条件付き承認/要修正）
- **実行可能なアクション**:
  - Critical: [即座に対応が必要]
  - Should: [次のスプリントで対応]
  - Nice: [技術的負債として追跡]
- **前提条件の明示**:
  - 評価が基づく前提
  - 確認が必要な事項
  - スコープ外の項目
- **トレードオフの分析**:
  - 品質と速度のバランス
  - Reviewer 2の評価との比較
  - 総合的な判断

### Reviewer 2 (reviewer-2)

Phase 1/2の成果物に対して別の視点からレビューを実施する。Reviewer 1とは異なる観点から評価し、包括的な品質保証を行う。

#### Task Approach

1. **Phase 1/2 の結果を収集**
   - Reviewer 1と共通の前提を理解
   - 異なる観点からの評価ポイントを特定
   - 相互検証を実施

2. **品質評価を実施**
   - Reviewer 1とは異なる観点から評価
   - 保守性、テスト容易性を評価
   - ドキュメントの完全性を確認

3. **統合分析を実施**
   - Reviewer 1の評価結果との比較
   - 合意点と相違点を特定
   - 統合された結論を導出

4. **簡潔で実行可能な出力を返す**
   - 品質評価のサマリー
   - Reviewer 1との統合結果
   - 最終的な推奨事項

#### Output Format

- **簡潔な結論**:
  - 品質評価の概要
  - Reviewer 1との合意点/相違点
  - 総合評価（承認/条件付き承認/要修正）
- **実行可能なアクション**:
  - Critical: [即座に対応が必要]
  - Should: [次のスプリントで対応]
  - Nice: [技術的負債として追跡]
- **前提条件の明示**:
  - 評価が基づく前提
  - Reviewer 1との共通理解
  - スコープ外の項目
- **トレードオフの分析**:
  - Reviewer 1の評価との比較
  - 統合された判断
  - 残存する検討事項

## 警告信号 - プロセスの遵守を促す

以下のような考えが浮かんだら、それはSTOPのサイン:
- 「Reviewer 1の結果を確認せずに結論を出す」
- 「矛盾はトレードオフとして無視しよう」
- 「大まかなレビューで十分」

**これらすべては: STOP。統合プロセスを見直せ。**

## 人間のパートナーの「やり方が間違っている」シグナル

**以下の方向転換に注意:**
- 「これらは矛盾していないか？」 - 統合が不十分
- 「前提条件は確認したか？」 - 前提が不明確
- 「具体的なアクションは？」 - 実行可能でない
- 「どちらが優先か？」 - 優先順位付けが不明確

**これらを見たら:** STOP。統合プロセスを見直せ。

## よくある言い訳

| 言い訳 | 現実 |
|--------|------|
| 「このフェーズだけで十分」 | 単一視点は盲点を生む。複数視点が本質を捉える。 |
| 「統合は後で」 | 後では文脈を失う。分析と同時に統合する。 |
| 「どちらかを採用」 | どちらかではなく、統合した最適解を導く。 |
| 「大まかでいい」 | 曖昧な計画は曖昧な実行を生む。 |

## クイックリファレンス

| フェーズ | 主要活動 | 成功基準 |
|-------|---------------|------------------|
| **Phase 1** | 担当領域の独立したスライスの分析 | 特定視点からの発見が明確 |
| **Phase 2** | 実装・詳細設計の実行 | 計画に基づく実装が完了 |
| **Phase 3** | 統合・レビュー・品質保証 | 実行可能な統合計画 |

## 分析が「不十分」を示した場合

分析結果が不十分と判断された場合:

1. 不足している視点を特定
2. 追加の分析を実施するか、既存の分析を深掘り
3. 前提条件の確認を徹底
4. 統合前に各視点の完全性を確保

**しかし:** 完璧な分析より、十分な情報での意思決定が重要。80/20で前進する。
