# Skill Creation - Phase 1: Design

## チームミッション

Skill CreationのPhase 1（要件定義・設計）を担当。スキル作成の土台を構築する。

**出力:** 設計結果は Phase 2（skill-creation-p2）に引き継がれる。

## Output Format

```
SUMMARY: [設計サマリー]
CLAIM: [スキルの目的と価値]
EVIDENCE: [要件、構造設計]
CONFIDENCE: [0.00-1.00]
RESULT:
## スキルの目的
- [目的]

## 使用タイミング
- [タイミング1]
- [タイミング2]

## ディレクトリ構造
[構造図]

## 機能範囲
- 含む: [...]
- 含まない: [...]
NEXT_STEP: Phase 2（skill-creation-p2）でSKILL.md作成
```

## When to Use

**このフェーズを使用する:**
- 新しいスキルの作成リクエストを受け取った時
- スキルの目的・範囲が不明確な状態
- 既存スキルの拡張・改良が必要
- チーム共有用スキルの開発

**このフェーズをスキップすべきでない場合:**
- 目的が曖昧なまま作成を進めようとしている
- スキル名が決まっていない
- 機能範囲の境界が定義されていない

## 警告信号 - プロセスの遵守を促す

| 警告信号 | 何をすべきか |
|---------|-------------|
| 「とりあえず作ってみる」 | 目的を明確に定義してから進める |
| 「名前は後で決める」 | スキル名規約に従い即座に決定 |
| 「必要なものは作成中に考える」 | 機能範囲を先に定義 |
| 「構造は適当でいい」 | 標準構造に従い設計 |
| 「descriptionは後で書く」 | 設計段階で説明文案を作成 |

## 人間のパートナーの「やり方が間違っている」シグナル

| シグナル | 意味 | 推奨アクション |
|---------|------|---------------|
| 「このスキルは何のためにあるの？」 | 目的が不明確 | 要件ヒアリングを実施 |
| 「他に似たスキルない？」 | 重複の可能性 | 既存スキルとの差分確認 |
| 「名前がわかりにくい」 | 命名不適切 | 目的を反映した名称に変更 |
| 「機能が多すぎる」 | スコープ過大 | 機能を分割・優先順位付け |
| 「いつ使うかわからない」 | 使用タイミング未定義 | 具体的な使用シナリオを特定 |

## よくある言い辞

| 言い辞 | なぜ危険か | 正しいアプローチ |
|-------|-----------|-----------------|
| 「まずコードを書いてから考える」 | 手戻りが大きい | 設計→実装の順序を守る |
| 「複雑な構造は必要ない」 | 拡張性不足 | 将来を見据えた構造設計 |
| 「名前は適当でいい」 | 検索性・一貫性低下 | 命名規約に従う |
| 「descriptionは短くていい」 | 用途が伝わらない | 1024文字以内で十分に記述 |
| 「他のスキルを参考にしなくていい」 | 標準から逸脱 | 既存スキルの構造を確認 |

## 鉄の掟

```
目的の不明確なスキルを作成開始しない
命名規約に違反する名前を採用しない
設計書なしにPhase 2に進まない
```
