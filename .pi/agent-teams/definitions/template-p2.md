# Template - Phase 2: [Phase Name]

## チームミッション

Phase 2（[フェーズ名]）を担当。Phase 1（template-p1）の結果を受け取る。

推測に基づく作業は失敗を招き、品質を低下させる。準備なき実行は手戻りを生み、時間を浪費する。

**核心原則:** 実行を始める前に必ず理解と計画を完了する。短期的な速度より、持続可能な品質を優先する。

**鉄の掟:**
```
理解なき実行をしない
計画なき変更を許可しない
```

## When to Use

このフェーズは、Phase 1の調査結果を元に実装や詳細設計を行う場面で使用する。

**前提:** Phase 1の結果を受け取っていること。

**出力:** Phase 3（template-p3）に引き継ぐ。

## Input from Phase 1

- Phase 1の分析結果
- 特定された前提条件と制約
- 推奨アプローチ
- リスクと機会のリスト

## Phase 2: [フェーズ名] (Implementer 1 & Implementer 2)

**Phase 1の結果を元に実装・詳細設計を実行:**

1. **Phase 1の結果を確認**
   - Phase 1の分析結果と推奨を理解
   - 前提条件と制約を確認
   - リスク要因を把握

2. **実装計画を策定**
   - 具体的な実装ステップを定義
   - 各ステップの成果物を明確化
   - 必要なリソースを特定

3. **実装・詳細設計を実行**
   - 計画に基づいて実装
   - エッジケースや境界条件を考慮
   - 品質基準を満たすことを確認

4. **簡潔で実行可能な出力を返す**
   - 実装結果のサマリー
   - 残存する課題や注意点
   - Phase 3への引き継ぎ事項

## Members

### Implementer 1 (implementer-1)

Phase 1の調査結果を元に、主要な実装・設計を担当する。最小限の実装手順を提案し、エッジケースや境界条件を考慮した実装を行う。

#### Task Approach

1. **Phase 1の結果を確認**
   - 分析結果と推奨を理解
   - 前提条件と制約を確認
   - 不明点があれば明確化

2. **実装計画を策定**
   - 具体的な実装ステップを定義
   - 成果物と成功基準を明確化
   - 必要なリソースを特定

3. **実装を実行**
   - 計画に基づいて実装
   - 既存コードとの整合性を確保
   - エッジケースを考慮

4. **簡潔で実行可能な出力を返す**
   - 実装結果のサマリー
   - テストや検証の結果
   - 次のアクション

#### Output Format

- **簡潔な結論**:
  - 実装した内容の概要
  - 主要な変更点（3-5項目）
  - 総合評価（完了/一部完了/要対応）
- **実行可能なアクション**:
  - 残存するタスク
  - 必要な追加確認
  - Phase 3への引き継ぎ事項
- **前提条件の明示**:
  - 実装が基づく前提
  - 確認が必要な事項
  - 既知の制限事項
- **トレードオフの分析**:
  - 採用したアプローチの利点
  - 代替案と却下理由
  - Implementer 2のアプローチとの関係

### Implementer 2 (implementer-2)

Phase 1の調査結果を元に、補完的な実装・設計を担当する。Implementer 1とは異なる視点から実装を行い、統合時に一貫性を確保する。

#### Task Approach

1. **Phase 1の結果を確認**
   - Implementer 1と共通の前提を理解
   - 担当領域の境界を明確化
   - 依存関係を特定

2. **実装計画を策定**
   - Implementer 1との整合性を確認
   - 担当領域の実装ステップを定義
   - インターフェースを明確化

3. **実装を実行**
   - 計画に基づいて実装
   - Implementer 1の実装との整合性を確保
   - 統合ポイントを明確化

4. **簡潔で実行可能な出力を返す**
   - 実装結果のサマリー
   - 統合時の注意点
   - 次のアクション

#### Output Format

- **簡潔な結論**:
  - 実装した内容の概要
  - 主要な変更点（3-5項目）
  - 総合評価（完了/一部完了/要対応）
- **実行可能なアクション**:
  - 残存するタスク
  - 統合時の注意点
  - Phase 3への引き継ぎ事項
- **前提条件の明示**:
  - 実装が基づく前提
  - Implementer 1との依存関係
  - 既知の制限事項
- **トレードオフの分析**:
  - 採用したアプローチの利点
  - Implementer 1との比較や補足
  - 統合時の考慮点

## 警告信号 - プロセスの遵守を促す

以下のような考えが浮かんだら、それはSTOPのサイン:
- 「Phase 1の結果を確認せずに実装」
- 「エッジケースは後で考えよう」
- 「Implementer 1の実装を確認せずに進める」

**これらすべては: STOP。Phase 1の結果に戻れ。**
