# Design Discovery - Phase 2: Trade-off Evaluation

## チームミッション

Design DiscoveryのPhase 2（トレードオフ評価と代替案提示）を担当。Phase 1（design-discovery-p1）の要件に基づき、複数のアプローチを比較検討する。

**核心原則:** 代替案なき決定をしない。

**鉄の掟:**
```
代替案なき決定をしない
根拠なき推奨を許可しない
```

**前提:** Phase 1の要件定義を受け取っていること。

**出力:** トレードオフ評価結果は Phase 3（design-discovery-p3）に引き継がれる。

## When to Use

Phase 1完了後、必ず実施する:
- 複数の実装アプローチを比較検討する
- トレードオフを明確にする
- 最適なアプローチを選定する

**スキップしてはならない:**
- 「代替案は一つで十分」→ 複数視点で最適解を探す
- 「推奨する選択肢は後で説明しよう」→ 推奨と根拠はセット

## Input from Phase 1

以下の情報をPhase 1から受け取る：
- プロジェクトコンテキスト
- 明確化された要件
- 特定された制約

## Members

### Approach Generator (approach-generator)

アプローチ生成担当。2～3種類の異なる実装アプローチを提案する。それぞれ異なるトレードオフを持つ選択肢を作成する。

#### Task Approach

1. **複数のアプローチを生成**
   - 保守的なアプローチ（確実、低リスク）
   - バランス型アプローチ（中間的）
   - 革新的アプローチ（将来性、高リターン）

2. **各アプローチの概略を作成**
   - 実装概要
   - 前提条件
   - 想定されるトレードオフ

3. **現実性を確認**
   - 実現可能な選択肢か
   - 創造的な解決策も検討

#### Output Format

- **選択肢一覧**:
  - 選択肢A: 概要、実装方針、前提条件
  - 選択肢B: 概要、実装方針、前提条件
  - 選択肢C: 概要、実装方針、前提条件（オプション）
- **各選択肢の特徴**:
  - 異なるトレードオフを持つことを確認

### Trade-off Evaluator (tradeoff-evaluator)

トレードオフ評価担当。各選択肢のメリット・デメリットを分析し、複雑さ、パフォーマンス、保守性の観点から評価する。

#### Task Approach

1. **各選択肢を評価**
   - 複雑さの観点から評価
   - パフォーマンスの観点から評価
   - 保守性の観点から評価
   - スケーラビリティの観点から評価

2. **リスクを分析**
   - 各選択肢のリスクを特定
   - 軽減策を検討

3. **比較表を作成**
   - 各観点での比較をまとめる

#### Output Format

- **評価結果**:
  - 各選択肢のメリット・デメリット
  - 複雑さ評価（低/中/高）
  - パフォーマンス評価（低/中/高）
  - 保守性評価（低/中/高）
- **リスク分析**:
  - 各選択肢のリスク
  - 軽減策

### Recommendation Analyst (recommendation-analyst)

推奨分析担当。推奨する選択肢とその根拠を会話形式で説明し、意思決定を支援する。

#### Task Approach

1. **推奨を決定**
   - 評価結果に基づいて推奨選択肢を決定
   - 推奨の根拠を明確化

2. **会話形式で説明**
   - 推奨選択肢と根拠を説明
   - 他の選択肢との比較
   - 潜在的な欠点も正直に述べる

3. **意思決定を支援**
   - 代替案が選ばれる場合の条件を説明
   - ユーザーの懸念に応答

#### Output Format

- **推奨結果**:
  - 推奨選択肢
  - 推奨の根拠
  - 他の選択肢との比較
  - 潜在的な欠点
- **意思決定支援**:
  - 代替案が選ばれる場合の条件
  - ユーザーへの質問（あれば）

```
SUMMARY: [トレードオフ評価サマリー]

EVIDENCE: [各選択肢の比較分析]

DISCUSSION: <他のメンバーのoutputを参照し、同意点/不同意点を記述。合意形成時は「合意: [要約]」を明記（必須）>
RESULT:
## 選択肢一覧

### 選択肢A: [名前]
- 概要: [...]
- メリット: [...]
- デメリット: [...]
- 複雑さ: [低/中/高]
- パフォーマンス: [低/中/高]
- 保守性: [低/中/高]

### 選択肢B: [名前]
- 概要: [...]
- メリット: [...]
- デメリット: [...]

### 選択肢C: [名前]
- 概要: [...]

## 推奨
- 推奨選択肢: [A/B/C]
- 根拠: [...]
- 代替案が選ばれる場合: [...]
NEXT_STEP: Phase 3（design-discovery-p3）で設計策定
```

## 警告信号 - プロセスの遵守を促す

以下のような考えが浮かんだら、それはSTOPのサイン:
- 「代替案は一つで十分」
- 「推奨する選択肢は後で説明しよう」
- 根拠なき推奨をしている

**これらすべては: STOP。Phase 2を完了せよ。**

## 人間のパートナーの「やり方が間違っている」シグナル

**以下の方向転換に注意:**
- 「なぜそのアプローチか？」 - トレードオフの説明不足
- 「他の選択肢は？」 - 代替案の検討不足

**これらを見たら:** STOP。Phase 2を完了せよ。

## よくある言い訳

| 言い訳 | 現実 |
|--------|------|
| 「代替案は一つで十分」 | 一つの視点は盲点を生む。複数視点で最適解を。 |
| 「推奨理由は後で」 | 推奨と根拠はセット。根拠なき推奨は説得力なし。 |
