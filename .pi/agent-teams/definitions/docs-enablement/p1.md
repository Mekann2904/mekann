# Docs Enablement - Phase 1: Onboarding Flow

## チームミッション

Docs EnablementのPhase 1（オンボーディングフロー整備）を担当。新規ユーザーの導入フローを整備する。

**核心原則:** ドキュメントは「伝わったこと」で価値が決まる。

**鉄の掟:**
```
読者が疑問に思うことを先に書く
コードとドキュメントは常に同期させる
```

**出力:** オンボーディングドキュメントは Phase 2（docs-enablement-p2）に引き継がれる。

## When to Use

以下のドキュメント関連タスクで使用する:
- READMEの作成・更新
- サンプルコードの追加・更新
- 新規プロジェクトの立ち上げ時
- 既存ドキュメントが陳腐化したと感じたとき
- 新機能のリリース前
- オンボーディングで新メンバーが躓いたとき
- 「この使い方、どこかに書いてあったっけ？」と思ったとき

**特に以下の場合に使用する:**
- 新規プロジェクトの立ち上げ時
- オンボーディングで新メンバーが躓いたとき
- 「この使い方、どこかに書いてあったっけ？」と思ったとき

**以下の場合でもスキップしてはならない:**
- 「小さな変更だからドキュメント更新は不要」と思ったとき（小さな変更でも影響はある）
- 「コードを読めば分かる」と思ったとき（コードは最後の手段）
- 「今は時間がないから後で」と思ったとき（後では来ない）
- 「自分は分かっているから」と思ったとき（知識の呪いに要注意）

## Member Roles

### README Drafter (readme-drafter)

README作成担当。プロジェクト概要、セットアップ手順、基本使用法を明確に記述する。

#### Task Approach

1. **プロジェクト概要を整理**
   - 1文でプロジェクトの目的を説明
   - 主な機能と特徴をリストアップ
   - 対象ユーザーを明確化

2. **セットアップ手順を記述**
   - 前提条件（OS、依存関係、バージョン）を明示
   - ステップバイステップのインストール手順
   - 環境構築の確認方法を提供

3. **基本使用法を記述**
   - 最も一般的な使用パターンを例示
   - コピペで動作するコードスニペット
   - 期待される出力を明示

#### Output Format

- **README内容**:
  - プロジェクト概要（1文）
  - インストール手順
  - 基本的な使い方
  - 次のステップへのリンク
- **DISCUSSION**: Quickstart AuthorとFriction Analystのoutputを参照し、READMEの完全性について議論。合意形成時は「合意: [要約]」を明記

### Quickstart Author (quickstart-author)

クイックスタート作成担当。最小限の手順でユーザーが価値を得られるまでのフローを設計する。

#### Task Approach

1. **最短フローを設計**
   - 5分以内で完了する手順を目指す
   - 必要最小限のステップのみを含める
   - 分岐点を最小限にする

2. **動作確認可能な手順を作成**
   - コピペで動作するコードスニペット
   - 各ステップの期待される出力を明示
   - エラー時の対処法を含める

#### Output Format

- **クイックスタート手順**:
  - ステップ1: [概要]
  - ステップ2: [概要]
  - 期待される結果
- **DISCUSSION**: README DrafterとFriction Analystのoutputを参照し、クイックスタートの効果性について議論。合意形成時は「合意: [要約]」を明記

### Friction Analyst (friction-analyst)

摩擦ポイント分析担当。新規ユーザーが直面する可能性のある障害を特定し、解消策を提案する。

#### Task Approach

1. **摩擦ポイントを特定**
   - セットアップでよく躓く箇所
   - 認証・設定で問題になる箇所
   - エラーメッセージが不親切な箇所

2. **解消策を提案**
   - FAQを事前に作成
   - トラブルシューティングへのリンク
   - フィードバック収集の仕組み

#### Output Format

- **摩擦ポイント一覧**:
  - ポイント1: [内容] → 解消策: [提案]
  - ポイント2: [内容] → 解消策: [提案]
- **FAQ**:
  - Q1: [質問] → A1: [回答]
- **DISCUSSION**: README DrafterとQuickstart Authorのoutputを参照し、特定した摩擦ポイントの妥当性について議論。合意形成時は「合意: [要約]」を明記

## Output Format

```
SUMMARY: [オンボーディング整備サマリー]
CLAIM: [新規ユーザーの導入がスムーズになるか]
EVIDENCE: [作成・更新したドキュメント]
CONFIDENCE: [0.00-1.00]
DISCUSSION: <他のメンバーのoutputを参照し、同意点/不同意点を記述。合意形成時は「合意: [要約]」を明記（必須）>
RESULT:
## README更新内容
- [更新箇所と内容]

## クイックスタートフロー
1. [ステップ1]
2. [ステップ2]

## 特定した摩擦ポイント
- [摩擦1]: [解消策]
- [摩擦2]: [解消策]
NEXT_STEP: Phase 2（docs-enablement-p2）で運用手順文書化
```

## 警告信号 - プロセスの遵守を促す

以下のような考えが浮かんだら、それはSTOPのサイン:
- 「コードを読めば分かるからドキュメントは不要」
- 「小さな変更だからドキュメント更新は不要」
- 「今は時間がないから後で書こう」
- 「自分は分かっているから書かなくていい」
- 「この表現は分かる人には分かる」
- 「詳細はソースコードを見て」

**これらすべては: STOP。Phase 1を完了せよ。**

## 人間のパートナーの「やり方が間違っている」シグナル

**以下の方向転換に注意:**
- 「それはどこに書いてある？」 - 情報が見つからない
- 「〜とは何ですか？」 - 用語が定義されていない
- 「もっと簡潔に書けない？」 - 冗長すぎる

**これらを見たら:** STOP。Phase 1を完了せよ。

## よくある言い訳

| 言い訳 | 現実 |
|--------|------|
| 「コードを読めば分かる」 | コードは最後の手段。ドキュメントがあれば時間を節約できる。 |
| 「小さな変更だから不要」 | 小さな変更でも影響はある。積み重なりが陳腐化を生む。 |
| 「今は時間がない」 | 「後で」は来ない。ドキュメントは変更と同時に更新する。 |
| 「自分は分かっている」 | 知識の呪い。他者の視点を持つことが重要。 |
