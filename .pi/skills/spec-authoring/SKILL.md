# spec.md作成支援スキル

対話的に質問を繰り返しながら、完全なドメイン仕様書（spec.md）を作成するためのガイド。

## 目的

- 漏れのない包括的な仕様書を作成する
- 不変条件・契約・境界条件を体系的に整理する
- ABDDワークフローと統合された仕様書を作成する

---

## 使用タイミング

以下の場合にこのスキルを読み込む:

- 新規機能・モジュールの仕様書を作成する
- 既存のspec.mdを見直し・更新する
- ABDD導入時に初期仕様書を作成する
- コードレビューで仕様の抜け漏れを発見した

---

## 使用方法

### questionツールによる対話フロー

`question`ツールを使用して、段階的に情報を収集します。

**Phase 1: 基本情報**
```typescript
question({
  question: "作成対象は何ですか？",
  options: [
    { label: "新規プロジェクト全体" },
    { label: "特定の機能・モジュール" },
    { label: "既存spec.mdの更新" }
  ]
})
```

**Phase 2: 不変条件**
```typescript
question({
  question: "このシステムにおいて「絶対に崩れてはいけない」ルールは何ですか？",
  options: [
    { label: "同じ入力には同じ出力を返す（冪等性）" },
    { label: "無効な状態になりえない" },
    { label: "エラー後もデータは破損していない" },
    { label: "リソースは必ず解放される" },
    { label: "その他（テキストで指定）" }
  ],
  multiple: true
})
```

**Phase 3: 契約**
```typescript
question({
  question: "戻り値の型・構造について、どのような約束がありますか？",
  options: [
    { label: "Result型（ok/errorの区別）" },
    { label: "特定のスキーマに従う" },
    { label: "null非許容" },
    { label: "配列の長さ制限あり" },
    { label: "該当なし" }
  ],
  multiple: true
})
```

## ベストプラクティス

### 不変条件を書く際の注意

- **具体的に**: 「適切に処理する」ではなく「3秒以内に応答する」
- **検証可能に**: テストで確認できる形で記述
- **網羅的に**: 正常系だけでなく異常系も考慮

### 契約を書く際の注意

- **型で表現**: 可能な限り型システムで強制
- **事前条件は最小限**: 呼び出し側の負担を減らす
- **事後条件は厳密に**: 呼び出し側が期待できる保証を明確に

### 境界条件を書く際の注意

- **現実的な値**: 理論上の最大値ではなく、実用上の制限
- **超過時の動作**: 単に「エラー」ではなく、具体的な対応を記述
- **理由を説明**: なぜその制限があるかをコメントで補足

## よくある質問

### Q: philosophy.mdとの違いは？

A: philosophy.mdは「価値観・優先順位」（主観的）、spec.mdは「不変条件・契約」（客観的）。philosophy.mdが「何を重視するか」を定義するのに対し、spec.mdは「何が常に成り立つべきか」を定義する。

### Q: 既存コードからspec.mdを逆生成できますか？

A: `abdd_spec_sync`ツールで一部自動生成可能。ただし、意図や背景情報は人間の入力が必要。

### Q: spec.mdは誰が更新する？

A: 設計変更を行った開発者が更新。ABDDレビュー時に乖離が検出された場合も更新が必要。

### Q: 小さい変更でもspec.mdを更新する必要がある？

A: 不変条件・契約・境界条件に影響しない変更（内部リファクタリング等）は不要。ただし、変更内容によっては見直しが推奨される。
