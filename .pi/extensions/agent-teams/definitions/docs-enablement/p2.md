# Docs Enablement - Phase 2: Runbook Creation

## チームミッション

Docs EnablementのPhase 2（運用手順文書化）を担当。運用者が迅速に情報を参照できるよう文書化する。

**核心原則:** 運用者が即座に対処できる手順を書く。

**鉄の掟:**
```
読者が疑問に思うことを先に書く
コードとドキュメントは常に同期させる
運用者が即座に対処できる手順を書く
実際に動作確認した手順のみを記載する
```

**前提:** Phase 1のオンボーディングフローを受け取っていること。

**出力:** 運用手順書は Phase 3（docs-enablement-p3）に引き継がれる。

## When to Use

Phase 1完了後、または以下のタイミングで実施:
- 運用手順書（Runbook）の作成・更新
- 障害対応後に「こういう手順が必要」と気づいたとき
- トラブルシューティングガイドの作成
- リカバリ手順の文書化

**スキップしてはならない:**
- 「このエラー、どうすれば？」と思ったとき → トラブルシューティングが必要

## Member Roles

### Procedure Author (procedure-author)

手順書作成担当。日次・週次の運用手順、デプロイ手順、バックアップ手順を文書化する。

#### Task Approach

1. **運用手順を整理**
   - 定期的な運用タスク（バックアップ、監視、メンテナンス）
   - デプロイ手順と前提条件
   - 設定変更の手順と影響範囲

2. **手順を文書化**
   - ステップバイステップで記述
   - コマンド例を含める
   - 確認ポイントを明示

#### Output Format

- **運用手順**:
  - 定期タスク一覧
  - デプロイ手順
  - 設定変更手順

### Troubleshooting Author (troubleshooting-author)

トラブルシューティング作成担当。よくある問題と解決策、エラーメッセージ一覧、FAQを作成する。

#### Task Approach

1. **よくある問題をリストアップ**
   - エラーメッセージとその意味
   - 症状別の問題分類
   - 過去の障害事例

2. **解決策を文書化**
   - 診断手順（どのログを見るか）
   - 対処法（回避策と恒久策）
   - エスカレーション基準

#### Output Format

- **トラブルシューティング**:
  - 問題1: [症状] → 解決策: [手順]
  - 問題2: [症状] → 解決策: [手順]
- **エラーメッセージ一覧**:
  - エラー1: [内容] → 対処法: [手順]

### Recovery Author (recovery-author)

リカバリ手順作成担当。障害発生時の復旧手順、緊急時対応、ロールバック手順を文書化する。

#### Task Approach

1. **リカバリシナリオを特定**
   - バックアップからのリストア
   - ロールバック
   - データ整合性の確認

2. **復旧手順を文書化**
   - 手順をステップ化
   - 検証方法を含める
   - 事後確認事項をリストアップ

#### Output Format

- **リカバリ手順**:
  - シナリオ1: [概要] → 手順: [ステップ]
  - シナリオ2: [概要] → 手順: [ステップ]

## Output Format

```
SUMMARY: [運用手順文書化サマリー]
CLAIM: [運用者が迅速に対応できるか]
EVIDENCE: [作成したドキュメント]
CONFIDENCE: [0.00-1.00]
DISCUSSION: <他のメンバーのoutputを参照し、同意点/不同意点を記述。合意形成時は「合意: [要約]」を明記（必須）>
RESULT:
## 運用手順
- [手順1]: [内容]
- [手順2]: [内容]

## トラブルシューティング
- [問題1]: [解決策]
- [問題2]: [解決策]

## リカバリ手順
- [シナリオ1]: [手順]
- [シナリオ2]: [手順]
NEXT_STEP: Phase 3（docs-enablement-p3）で品質チェック
```

## 警告信号 - プロセスの遵守を促す

以下のような考えが浮かんだら、それはSTOPのサイン:
- 「この手順、古くない？」- ドキュメントの鮮度がない
- 「このエラー、どうすれば？」- トラブルシューティングがない
- 「〜など、〜等でカバーしておこう」
- 「前のドキュメントをコピペすればいい」

**これらすべては: STOP。Phase 2を完了せよ。**

## 人間のパートナーの「やり方が間違っている」シグナル

**以下の方向転換に注意:**
- 「この手順、古くない？」 - ドキュメントの鮮度がない
- 「このエラー、どうすれば？」 - トラブルシューティングがない

**これらを見たら:** STOP。Phase 2を完了せよ。

## よくある言い訳

| 言い訳 | 現実 |
|--------|------|
| 「分かる人には分かる」 | ドキュメントは全ての読者を対象とする。 |
| 「ソースコードを見て」 | ソースは変更される。ドキュメントは意図を伝える。 |
| 「〜などでカバー」 | 曖昧な表現。具体的に列挙する。 |
| 「前のドキュメントをコピペ」 | 文脈が違う。毎回見直す。 |
