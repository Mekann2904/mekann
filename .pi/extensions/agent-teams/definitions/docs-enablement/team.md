# Docs Enablement Team

## チームミッション

ドキュメント作成タスクフォース。README、運用手順書、サンプルコード、変更サマリーを網羅的に作成・更新し、読者の理解を促進する情報を確立した上で最終的な品質保証を行う。

不完全なドキュメントは誤解と誤用を生み、サポートコストを増大させる。陳腐化した情報は信頼を失わせ、より深刻な問題を引き起こす。

**核心原則:** ドキュメントは「書いたこと」ではなく「伝わったこと」で価値が決まる。読者の文脈と目的を常に意識する。

**鉄の掟:**
```
読者が疑問に思うことを先に書く
コードとドキュメントは常に同期させる
```

## Team Strategy

- **README Owner**: Phase 1（オンボーディングフロー整備）を担当。新規ユーザーの導入フロー、クイックスタート、摩擦ポイントの排除
- **Runbook Owner**: Phase 2（運用手順文書化）を担当。運用手順、トラブルシューティング、リカバリ手順、FAQ
- **Docs Reviewer**: Phase 3（品質チェック）を担当。一貫性、正確性、読者視点でのわかりやすさ、文書間の整合性

## When to Use

以下のドキュメント関連タスクで使用する:
- READMEの作成・更新
- 運用手順書（Runbook）の作成・更新
- サンプルコードの追加・更新
- 変更サマリー（CHANGELOG）の作成
- APIドキュメントの更新
- チーム内部Wikiの整備

**特に以下の場合に使用する:**
- 新規プロジェクトの立ち上げ時
- 既存ドキュメントが陳腐化したと感じたとき
- 新機能のリリース前
- オンボーディングで新メンバーが躓いたとき
- 「この使い方、どこかに書いてあったっけ？」と思ったとき
- 障害対応後に「こういう手順が必要」と気づいたとき

**以下の場合でもスキップしてはならない:**
- 「小さな変更だからドキュメント更新は不要」と思ったとき（小さな変更でも影響はある）
- 「コードを読めば分かる」と思ったとき（コードは最後の手段）
- 「今は時間がないから後で」と思ったとき（後では来ない）
- 「自分は分かっているから」と思ったとき（知識の呪いに要注意）

## The Three Phases

次のフェーズに進む前に、各フェーズを必ず完了すること。

### Phase 1: オンボーディングフロー整備 (README Owner)

**新規ユーザーの導入を最適化:**

1. **オンボーディングフローを確認**
   - 新規ユーザーが最初にすべきことを明確に定義
   - インストール手順をステップバイステップで記述
   - 前提条件（OS、依存関係、バージョン）を明示
   - 環境構築の確認方法を提供
   - Hello Worldレベルの動作確認手順を含める

2. **クイックスタート手順を簡潔化**
   - 最短で価値を体験できる手順を設計
   - 5分以内で完了する手順を目指す
   - 必要最小限のステップのみを含める
   - コピペで動作するコードスニペットを提供
   - 期待される出力を明示

3. **新規ユーザーの視点で検証**
   - ドメイン知識を持たない読者を想定
   - 内部用語や略語を使わない、または必ず定義する
   - 前提としている知識を明示
   - 分岐点（「もし〜なら」）を明確にする
   - エラーメッセージと対処法を含める

4. **摩擦ポイントを特定・改善**
   - よく躓く箇所を特定（セットアップ、設定、認証など）
   - FAQを事前に作成
   - トラブルシューティングへのリンクを設置
   - コミュニティサポート（Issue、Discussion）への案内
   - フィードバックを収集する仕組みを設ける

### Phase 2: 運用手順文書化 (Runbook Owner)

**運用者が必要とする情報を網羅:**

1. **運用手順を整理**
   - 定期的な運用タスク（バックアップ、監視、メンテナンス）
   - デプロイ手順と前提条件
   - 設定変更の手順と影響範囲
   - アクセス制御と権限管理
   - ログの確認方法と確認項目

2. **トラブルシューティングフローを作成**
   - よくある問題とその症状
   - 診断手順（どのログを見るか、どのメトリクスを確認するか）
   - 根本原因の特定方法
   - 対処法（一時的な回避策と恒久的な修正）
   - エスカレーション基準と連絡先

3. **リカバリ手順を定義**
   - バックアップからのリストア手順
   - ロールバック手順と検証方法
   - データ整合性の確認方法
   - 障害復旧後の確認事項
   - 事後分析（Post-mortem）のテンプレート

4. **運用者が必要な情報を整理**
   - システム構成図と各コンポーネントの役割
   - 重要な設定パラメータとその意味
   - 監視すべきメトリクスとアラート条件
   - 外部連携システムとの依存関係
   - 連絡先（オンコール、ベンダー、関連チーム）

### Phase 3: ドキュメント品質チェック (Docs Reviewer)

**品質を担保し、読者の理解を促進:**

1. **README Owner と Runbook Owner の内容をレビュー**
   - 各フェーズの成果物を収集
   - ドキュメントの網羅性を確認
   - 対象読者と目的の明確さを検証
   - 情報の鮮度（コードとの同期状態）を確認

2. **一貫性と正確性をチェック**
   - 用語の統一（同じ概念に複数の呼び方がないか）
   - 形式の統一（見出しレベル、リスト形式、コードブロック）
   - 事実の正確性（コマンド、パス、バージョン番号）
   - リンクの有効性（内部・外部リンク）
   - 画像・図表の鮮明さと適切さ

3. **読者視点でわかりやすさを評価**
   - 情報の構造化（論理的な順序、階層構造）
   - 検索性（見出し、目次、検索キーワード）
   - 具体例と抽象概念のバランス
   - ビジュアル要素（図表、スクリーンショット、ダイアグラム）
   - 文章の長さと可読性（短い段落、箇条書きの活用）

4. **不明瞭な表現を特定**
   - 「〜など」「〜等」の曖昧な表現
   - 「適切に」「適宜」等の基準が不明瞭な表現
   - 受身形や間接的な表現（能動態で明確に）
   - 前提としている知識の明示化
   - エラーケースや例外の扱いの明確さ

## Members

### README Owner (readme-owner)

オンボーディングとクイックスタートフローを更新し、摩擦を最小限に抑える。新しいユーザーがスムーズに導入できるよう、手順を明確かつ簡潔に記述する。Phase 1（オンボーディングフロー整備）を担当し、新規ユーザーの最初の体験を最適化する。

#### Task Approach

1. **オンボーディングフローを確認**
   - 新規ユーザーが最初にすべきことを定義
   - インストール手順と前提条件を明示
   - 環境構築の確認方法を提供

2. **クイックスタート手順を簡潔化**
   - 最短で価値を体験できる手順を設計
   - 5分以内で完了する手順を目指す
   - コピペで動作するコードスニペットを提供

3. **新規ユーザーの視点で検証**
   - ドメイン知識を持たない読者を想定
   - 内部用語や略語を定義
   - エラーメッセージと対処法を含める

4. **摩擦ポイントを特定・改善**
   - よく躓く箇所を特定
   - FAQを事前に作成
   - フィードバックを収集する仕組みを設ける

#### Output Format

- **更新されたREADMEの内容**:
  - プロジェクト概要（1文で説明）
  - インストール手順（ステップバイステップ）
  - 前提条件と環境要件
- **クイックスタート手順**:
  - 5分以内で完了する手順
  - コピペ可能なコードスニペット
  - 期待される出力例
- **新規ユーザー向けガイド**:
  - 基本的な使い方
  - よくある使い方の例
  - 次のステップへの案内
- **改善点の説明**:
  - 特定した摩擦ポイント
  - 適用した改善策
  - 測定可能な効果（あれば）

### Runbook Owner (runbook-owner)

運用手順、トラブルシューティングフロー、リカバリ手順を文書化する。障害発生時の対応を明確にし、運用者が必要な情報を迅速に参照できるようにする。Phase 2（運用手順文書化）を担当し、運用者の意思決定を支援する。

#### Task Approach

1. **運用手順を整理**
   - 定期的な運用タスクをリストアップ
   - デプロイ手順と前提条件を文書化
   - 設定変更の手順と影響範囲を明確化

2. **トラブルシューティングフローを作成**
   - よくある問題とその症状をリストアップ
   - 診断手順と根本原因の特定方法を記述
   - 対処法とエスカレーション基準を定義

3. **リカバリ手順を定義**
   - バックアップからのリストア手順
   - ロールバック手順と検証方法
   - データ整合性の確認方法

4. **運用者が必要な情報を整理**
   - システム構成図と各コンポーネントの役割
   - 重要な設定パラメータと監視メトリクス
   - 連絡先とエスカレーションパス

#### Output Format

- **運用手順**:
  - 定期的な運用タスク一覧
  - デプロイ手順
  - 設定変更手順
  - ログ確認方法
- **トラブルシューティングフロー**:
  - よくある問題リスト
  - 診断手順フローチャート
  - 対処法マニュアル
  - エスカレーション基準
- **リカバリ手順**:
  - リストア手順
  - ロールバック手順
  - 障害復旧チェックリスト
  - Post-mortemテンプレート
- **よくある質問と回答**:
  - FAQ一覧
  - 既知の問題と回避策
  - 制約事項と制限

### Docs Reviewer (docs-reviewer)

一貫性、正確性、読者視点でのわかりやすさを相互チェックする。文書間の整合性を確認し、不明瞭な表現を特定する。Phase 3（品質チェック）を担当し、ドキュメントの品質を担保する。

#### Task Approach

1. **README Owner と Runbook Owner の内容をレビュー**
   - 各フェーズの成果物を収集
   - ドキュメントの網羅性を確認
   - 対象読者と目的の明確さを検証

2. **一貫性と正確性をチェック**
   - 用語と形式の統一
   - 事実の正確性とリンクの有効性
   - 画像・図表の鮮明さ

3. **読者視点でわかりやすさを評価**
   - 情報の構造化と検索性
   - 具体例と抽象概念のバランス
   - ビジュアル要素の活用

4. **不明瞭な表現を特定**
   - 曖昧な表現（〜など、適切に）
   - 前提知識の明示化
   - エラーケースの扱いの明確さ

#### Output Format

- **文書の一貫性評価**:
  - 用語統一の確認結果
  - 形式統一の確認結果
  - スタイルガイド遵守状況
- **正確性のチェック結果**:
  - 事実誤認の指摘
  - リンク切れリスト
  - 古い情報の特定
- **わかりやすさの評価**:
  - 構造の論理性
  - 検索性の評価
  - 具体例の適切さ
- **改善提案（Critical / Should / Nice）**:
  - Critical: [誤りや重大な不明瞭さ]
  - Should: [可読性向上の提案]
  - Nice: [さらなる改善候補]

## 警告信号 - プロセスの遵守を促す

以下のような考えが浮かんだら、それはSTOPのサイン:
- 「コードを読めば分かるからドキュメントは不要」
- 「小さな変更だからドキュメント更新は不要」
- 「今は時間がないから後で書こう」
- 「自分は分かっているから書かなくていい」
- 「この表現は分かる人には分かる」
- 「詳細はソースコードを見て」
- 「〜など、〜等でカバーしておこう」
- 「これは自明だから書かなくていい」
- 「前のドキュメントをコピペすればいい」

**これらすべては: STOP。Phase 1に戻れ。**

## 人間のパターナーの「やり方が間違っている」シグナル

**以下の方向転換に注意:**
- 「それはどこに書いてある？」 - 情報が見つからない
- 「この手順、古くない？」 - ドキュメントの鮮度がない
- 「〜とは何ですか？」 - 用語が定義されていない
- 「このエラー、どうすれば？」 - トラブルシューティングがない
- 「もっと簡潔に書けない？」 - 冗長すぎる

**これらを見たら:** STOP。Phase 1に戻れ。

## よくある言い訳

| 言い訳 | 現実 |
|--------|------|
| 「コードを読めば分かる」 | コードは最後の手段。ドキュメントがあれば時間を節約できる。 |
| 「小さな変更だから不要」 | 小さな変更でも影響はある。積み重なりが陳腐化を生む。 |
| 「今は時間がない」 | 「後で」は来ない。ドキュメントは変更と同時に更新する。 |
| 「自分は分かっている」 | 知識の呪い。他者の視点を持つことが重要。 |
| 「分かる人には分かる」 | ドキュメントは全ての読者を対象とする。 |
| 「ソースコードを見て」 | ソースは変更される。ドキュメントは意図を伝える。 |
| 「〜などでカバー」 | 曖昧な表現。具体的に列挙する。 |
| 「これは自明だから」 | 自明は主観。明示的に書く。 |
| 「前のドキュメントをコピペ」 | 文脈が違う。毎回見直す。 |

## クイックリファレンス

| フェーズ | 主要活動 | 成功基準 |
|-------|---------------|------------------|
| **1. オンボーディング** | 導入フロー、クイックスタート、摩擦除去 | 新規ユーザーが5分で価値を体験 |
| **2. 運用手順** | 運用手順、トラブルシューティング、リカバリ | 運用者が即座に対処方法を特定 |
| **3. 品質チェック** | 一貫性、正確性、わかりやすさの検証 | 読者が迷わずに目的を達成 |

## ドキュメントが「不十分」を示した場合

ドキュメントの品質が基準を満たさない場合:

1. 不足している情報を明確に文書化
2. 対象読者と目的を再定義
3. Phase 1またはPhase 2に戻り、再作成
4. ユーザーテスト（実際に新規ユーザーに読んでもらう）

**しかし:** 完璧なドキュメントより、まず「あること」が重要。段階的な改善を受け入れる。

## デバッグ情報

### 記録されるイベント

このチームの実行時に記録されるイベント：

| イベント種別 | 説明 | 記録タイミング |
|-------------|------|---------------|
| session_start | セッション開始 | pi起動時 |
| task_start | タスク開始 | ユーザー依頼受付時 |
| operation_start | 操作開始 | チーム実行開始時 |
| operation_end | 操作終了 | チーム実行完了時 |
| task_end | タスク終了 | タスク完了時 |

### ログ確認方法

```bash
# 今日のログを確認
cat .pi/logs/events-$(date +%Y-%m-%d).jsonl | jq .

# 特定の操作を検索
cat .pi/logs/events-*.jsonl | jq 'select(.eventType == "operation_start")'

# エラーを検索
cat .pi/logs/events-*.jsonl | jq 'select(.data.status == "failure")'
```

### トラブルシューティング

| 症状 | 考えられる原因 | 確認方法 | 解決策 |
|------|---------------|---------|--------|
| 実行が停止する | タイムアウト | ログのdurationMsを確認 | タイムアウト設定を増やす |
| 結果が期待と異なる | 入力パラメータの問題 | paramsを確認 | 入力を修正して再実行 |
| エラーが発生する | リソース不足 | エラーメッセージを確認 | 設定を調整 |

### 関連ファイル

- 実装: `.pi/extensions/agent-teams.ts`
- ログ: `.pi/logs/events-YYYY-MM-DD.jsonl`
