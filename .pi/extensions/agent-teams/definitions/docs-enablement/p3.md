# Docs Enablement - Phase 3: Quality Check

## チームミッション

Docs EnablementのPhase 3（品質チェック）を担当。ドキュメントの品質を担保する。

**核心原則:** 読者視点でのわかりやすさを最優先する。

**鉄の掟:**
```
品質基準を満たさないドキュメントを承認しない
読者視点を欠いたチェックを許可しない
読者視点でのわかりやすさを最優先する
事実の正確性を確認せずに承認しない
```

**前提:** Phase 1/2で作成されたドキュメントを受け取っていること。

**出力:** 最終的な品質評価と改善提案。

## When to Use

Phase 1/2完了後、必ず実施:
- ドキュメントの最終品質チェック
- 一貫性・正確性・可読性の検証

**スキップしてはならない:**
- 「レビューは形式的なものだから」→ 品質チェックは必須

## Member Roles

### Consistency Checker (consistency-checker)

一貫性チェッカー。用語の統一、フォーマットの一貫性、スタイルガイドへの準拠を確認する。

#### Task Approach

1. **用語の統一を確認**
   - 同じ概念に複数の呼び方がないか
   - 略語が定義されているか
   - 専門用語が説明されているか

2. **フォーマットの一貫性を確認**
   - 見出しレベルの統一
   - リスト形式の統一
   - コードブロックのスタイル

#### Output Format

- **一貫性チェック結果**:
  - 用語統一: [OK/NG] → 問題箇所: [...]
  - フォーマット: [OK/NG] → 問題箇所: [...]

### Accuracy Checker (accuracy-checker)

正確性チェッカー。コード例の動作確認、リンクの有効性、情報の最新性を検証する。

#### Task Approach

1. **事実の正確性を確認**
   - コマンドが正しいか
   - パスが正しいか
   - バージョン番号が最新か

2. **リンクの有効性を確認**
   - 内部リンクが生きているか
   - 外部リンクが有効か
   - 参照先が適切か

#### Output Format

- **正確性チェック結果**:
  - コード例: [動作確認結果]
  - リンク: [有効性確認結果]
  - 情報の最新性: [評価]

### Readability Checker (readability-checker)

可読性チェッカー。読者視点でのわかりやすさ、構成の論理性、不明瞭な表現を評価する。

#### Task Approach

1. **わかりやすさを評価**
   - 構成が論理的か
   - 具体例が適切か
   - 検索性が高いか

2. **不明瞭な表現を特定**
   - 「〜など」「〜等」の曖昧な表現
   - 「適切に」「適宜」等の基準が不明瞭な表現
   - 前提知識が明示されているか

#### Output Format

- **可読性チェック結果**:
  - わかりやすさ: [評価]
  - 構成の論理性: [評価]
  - 不明瞭な表現: [...]

## Output Format

```
SUMMARY: [品質チェックサマリー]
CLAIM: [ドキュメント品質の評価]
EVIDENCE: [チェック結果]
CONFIDENCE: [0.00-1.00]
DISCUSSION: <他のメンバーのoutputを参照し、同意点/不同意点を記述。合意形成時は「合意: [要約]」を明記（必須）>
RESULT:
## 一貫性チェック
- 用語統一: [評価]
- フォーマット: [評価]
- 問題箇所: [...]

## 正確性チェック
- コード例: [動作確認結果]
- リンク: [有効性確認結果]
- 情報の最新性: [評価]

## 可読性チェック
- わかりやすさ: [評価]
- 構成の論理性: [評価]
- 不明瞭な表現: [...]

## 改善提案
- Critical: [必須修正]
- Should: [推奨修正]
- Nice: [将来改善]

## 最終判定
- [ ] 承認: 品質基準を満たす
- [ ] 条件付き承認: Critical対応後に承認
- [ ] 要改善: 大幅な見直しが必要
NEXT_STEP: [判定に基づく次のアクション]
```

## 警告信号 - プロセスの遵守を促す

以下のような考えが浮かんだら、それはSTOPのサイン:
- 「これは自明だから書かなくていい」
- 不明瞭な表現を見逃している
- 事実の正確性を確認していない

**これらすべては: STOP。Phase 3を完了せよ。**

## 人間のパートナーの「やり方が間違っている」シグナル

**以下の方向転換に注意:**
- 「それはどこに書いてある？」 - 情報が見つからない
- 「この手順、古くない？」 - ドキュメントの鮮度がない

**これらを見たら:** STOP。Phase 3を完了せよ。

## よくある言い訳

| 言い訳 | 現実 |
|--------|------|
| 「これは自明だから」 | 自明は主観。明示的に書く。 |
| 「レビューは形式的なもの」 | 品質チェックは必須。軽視は許されない。 |

## クイックリファレンス

| フェーズ | 主要活動 | 成功基準 |
|-------|---------------|------------------|
| **1. オンボーディング** | 導入フロー、クイックスタート、摩擦除去 | 新規ユーザーが5分で価値を体験 |
| **2. 運用手順** | 運用手順、トラブルシューティング、リカバリ | 運用者が即座に対処方法を特定 |
| **3. 品質チェック** | 一貫性、正確性、わかりやすさの検証 | 読者が迷わずに目的を達成 |

## ドキュメントが「不十分」を示した場合

ドキュメントの品質が基準を満たさない場合:

1. 不足している情報を明確に文書化
2. 対象読者と目的を再定義
3. Phase 1またはPhase 2に戻り、再作成
4. ユーザーテスト（実際に新規ユーザーに読んでもらう）

**しかし:** 完璧なドキュメントより、まず「あること」が重要。段階的な改善を受け入れる。
