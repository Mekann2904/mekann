# Changelog

この project は `0.x` の間、厳密な semantic versioning を保証しません。user-facing な変更、設定変更、安全境界、migration note をここに記録します。

## Unreleased

- subagent 子 Pi 起動の sh 文字列結合を argv 化し、シェルインジェクション経路を `pi-session.ts` と統一（issue #149）。`KittyController.buildChildScript` が model id / thinking / agent path / initial message を POSIX 単一引用符で `sh -lc` 1 本に結合していた設計（`pi-session.ts` が放棄した「シェル文字列で状態構築」パターンと同根）を廃止。`buildChildTokens` が各 pi オプション（`--model` / `--thinking` / `-e` / `@file`）を独立 argv トークンで組み立て、`PI_SUBAGENT_*` マーカーは kitty `--env` で渡し、内容を含まない静的 wrapper（`sh -c <wrapper> pi-subagent <pi tokens>`、`"$@"` で転送）のみを sh に渡すように変更。バッククォート / `$` / 単一引用符を含む system prompt・model id・message が再パースされる余地を排除。あわせて `export PATH=...` による PATH 上書き（IC-111）と `exec "${SHELL:-sh}"` の残留（IC-112）を廃止し `--copy-env` + kitty `--hold` で pane を保持し、子 Pi への `--allow-remote-control` 常時付与も廃止（IC-110）。対象: `mekann/autonomy/subagent/kittyControl.ts`（`shellQuote`/`buildChildScript` 廃止、`buildChildTokens`/`CHILD_WRAPPER` 新設）、`kittyControl.test.ts`（`--env` 形式へ更新）、`kittyControl.argv.test.ts`（新設: argv 構築回帰テスト）。
- shell-mode terminal action の `SHELL` をホワイトリスト検証（issue #149, IC-105/138）。`process.env.SHELL` を無検証で `spawnSync` に渡していたため、`SHELL` が `/tmp/malware` 等を指すと全 shell-mode action が攻撃者制御バイナリを起動しえた。`resolveShell` が `/bin/`・`/usr/bin/` プレフィックスのみ許可し、それ以外（相対名・`..` トラバーサル含む）は `/bin/sh` へフォールバックするように変更。対象: `mekann/utils/terminal/actions.ts`（`resolveShell` 新設、`terminalActionArgv` が使用）、`mekann/utils/terminal/action-runner.ts`（`defaultSpawnAction` が使用）、`mekann/utils/terminal/actions.test.ts`（新設）。
- kitty `renderImage` の `spawnSync` を非同期 `spawn` 化（issue #149, IC-088）。`kitten icat` が数百 ms かかる画像描印でイベントループをブロックしていたのを、stdio 継承のまま `spawn` + `close` 待ちに変更しノンブロッキング化。対象: `mekann/utils/terminal/kitty/control.ts`、`control.test.ts`（`spawn` モックで非同期パスを検証）。
- `/issue <number>` が子issueを持たない通常の(leaf)issueで何も開かない不具合を修正。従来 `/issue <number>` は orchestrate 専用にハードコードされており(issue #71)、親PRD/epic でない issue では `nothing to orchestrate` と出力して worktree を開かなかった。インタラクティブリストの単一選択と同じスマート判定に統一: 親issue(子issueあり)なら orchestrate、そうでなければ worktree + Work Pi を直接起動する。親issueの orchestrate 挙動は後方互換。対象: `mekann/utils/issue/args.ts`(`orchestrate` mode → `open` mode)、`mekann/utils/issue/cli.ts`(`runOpen` 追加、スマート dispatch)、`mekann/utils/issue/extension.ts`(usage 表示更新)、`mekann/utils/issue/args.test.ts`(新設)。
- voice-notify の発話タイミングを `agent_end` から `message_end` に変更し、`<voice>` を含むアシスタントメッセージが完了した瞬間に発話するようにした。従来はエージェントのターン全体が終了するまでキューに蓄積し最後に1回まとめて `say` を呼んでいたため、処理が停止しないと発話されないように見えていた。また並走する `say` プロセスで音声が割れるのを防ぐため、発話を直列化するキュー（`isSpeaking` ロック + コールバック駆動の drain）を追加。1メッセージ内の複数 `<voice>` は従来通り `。` で結合して1発話とする。対象: `mekann/utils/voice-notify/index.ts`（`speak`/`voiceQueue`/`agent_end` 削除、`speechQueue`/`drainSpeechQueue`/`enqueueSpeech` 追加）、`mekann/utils/voice-notify/index.test.ts`（`node:child_process` の制御可能モックで直列キューの振る舞いを検証）。
- cache-friendly-prompt レポートの output-gate 削減指標を修正。従来の `stub化率` は `(totalBytes - threshold×件数) / totalBytes` で threshold (48 KiB) を baseline にする proxy だった。外部化対象は threshold 超の出力のみのため、threshold 付近（48–55 KiB）の出力が多いと `stub化率` が 0% に潰れ、過小評価していた。output-gate が stub の実 inline bytes を記録するよう修正し（`recording.ts` の summary に `, stub N bytes` を追加）、真の `(totalBytes - totalStubBytes) / totalBytes` を `inline削減率` として主指標化。古いイベントは preview 既定値 (8 KiB) で補完し後方互換を維持。`stub化率` は参考値（`stub化率（閾値超過削減率）`）として残存。対象: `mekann/context/{recording.ts,output-gate/{store,controller}.ts}`、`mekann/core/cache-friendly-prompt/{outputGateSavings,report,coverage.test}.ts`。
- `totalPromptTokenEstimate` が実 input tokens でないことを明記。この値は `extractTextFromProviderPayload()` が provider payload から抽出したテキスト（content/text/system 等の preferred keys）の文字数ベース推定であり、tool schema・JSON 構造・数値フィールドを含まないため provider 報告の `inputTotalTokens` より常に小さくなる。`extract.ts` に doc comment、`request-snapshot.ts` の該当行に注記、report glossary に `totalPromptTokenEstimate` エントリを追加。
- Work Pi のモデル設定を `modes`（Collaboration Modes）に一元化。review-fixer と issue の Work Pi モデル・thinking が feature 固有タブに散在していたのを、`modes` の Work Pi プロファイル（`models.review_fix` / `thinking.review_fix` / `models.issue` / `thinking.issue`）に統合した。設定エディタの「Collaboration Modes」タブで全 Work Pi モデルを一覧管理できる。
  - **migration note**: 従来 `features.review-fixer.model` / `features.review-fixer.reasoningEffort` に設定していた値は無視されるようになった。`features.modes.models.review_fix` / `features.modes.thinking.review_fix` に再設定すること。
  - 新設 `issue` プロファイルで Issue Work Pi のモデルを初めて指定可能に（未設定なら pi デフォルト）。`launchPiSessionInKittySplit` が `--model provider/id` / `--thinking` を pi に渡し、autopilot / orchestration / bulk の全起動経路が modes から解決して渡す。対象: `mekann/safety/modes/{utils,settingsSchema,index}.ts`、`mekann/autonomy/review-fixer/{settingsSchema,settingsLoader}.ts`、`mekann/utils/terminal/pi-session.ts`、`mekann/utils/issue/orchestration/{issueModel.ts(新設),autopilot/extension.ts,extension.ts,cli.ts}`。
- `/issue` と `/issue-autopilot` で新しい Issue Work Pi が起動しない致命的不具合を修正（真因）。`launchPiSessionInKittySplit` が node/pi の引数を `JSON.stringify` でクォートした 1 本の `sh -lc "..."` コマンド文字列に結合していたため、`--append-system-prompt` の内容に含まれるバッククォート（マークダウンのコードフェンス例 `` `demote_to_ready_for_human` `` `` `issue_comment` ``）が sh のダブルクォート内で**コマンド置換として発火**し、`command not found` / `unmatched '` で pi が起動前に異常終了していた。`kitten @ launch` は末尾の argv を直接実行できるため、`node` + `pi` 引数を**生 argv 配列**（各トークンが独立 argv）で渡すように変更し、シェル再パスを完全に排除。対象: `mekann/utils/terminal/pi-session.ts`（`quoteShell` 廃止、argv 直接渡し）。
- `/issue` と `/issue-autopilot` が起動済みの Issue Work Pi pane を正しく検出できない不具合も修正。pi の対話モードが初期化を完了すると kitty のウィンドウタイトルを `π - <name> - <cwd>` に上書きするため、従来の先頭一致検出（`^Issue #N`）がマッチしなくなる問題だった。検出を env マーカー（`MEKANN_ISSUE_PI` / `MEKANN_AUTOPILOT_CHILD` / `MEKANN_ORCHESTRATION_CHILD`）ベースに変更（env は起動瞬間に設定され pi に上書きされないため初期化レースに強い）。タイトルはフォールバックとして残存。対象: `mekann/utils/terminal/kitty/control.ts`。
- `scripts/prepush-parallel.sh` の同時実行ジョブ数を `PREPUSH_MAX_JOBS`（デフォルト3）で制限し、テスト実行時のCPU過負荷を解消。従来は14ジョブを無制限に並列起動し、8コア環境で load avg 14超・node プロセス57個に達してコンテキストスイッチのスラッシングを起こしていた。制限後はピーク負荷を load avg 約11・プロセス15個に削減しつつテスト通過時間も短縮。CI や別コア数環境では環境変数で調整可能。: `output-gate/redact.ts` (shim), `isSafeCommand` (alias と re-export), `Mode` 型 alias, `CacheFriendlyPromptConfig.includeBaseSystemPromptInStableHash`。これらは後方互換のために残されていたが、非 test ファイルの呼び出し元が 0 件になっていた。`redactSecrets` のテストは実体のある `tool-output/redact.ts` を直接参照するよう移行済み。
- OSS 利用者向けの入口ドキュメントを追加。
- installation、configuration、architecture、support、security、contribution 導線を整備。
- GitHub issue template を追加。
