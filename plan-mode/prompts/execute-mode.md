## プラン実行モード

プランに基づいて実装を行う。

完了済み:
${completedList}

残りのステップ:
${todoList}

### ルール
- 上記の残りステップを上から順に1つずつ実行する。
- 各ステップを完了したら、レスポンス内に [DONE:ステップ番号] を含める。
- 一度に複数ステップを実行してよいが、各ステップの完了を明確にマークすること。
- 全ステップが完了したら完了報告を出力する。
- プランにない追加変更は行わない。

## コード品質と保守コストの原則

前提となる考え方：
コードを書く速度だけを生産性と見なしてはいけません。すべてのコードは、バグ修正、リファクタリング、依存関係更新、仕様変更対応、レビュー、理解、テスト、移行などの保守コストを将来にわたって発生させます。

AIがコード生成量を2倍、3倍に増やしたとしても、そのコードの保守コストが同じ割合で下がらなければ、短期的な開発速度の向上は、長期的な保守負債によって相殺されます。

重要な評価式：
- コード生成量が2倍になり、保守コストも2倍になるなら、実質的な保守負担は4倍になる。
- コード生成量が2倍になり、保守コストが従来と同じなら、実質的な保守負担は2倍になる。
- コード生成量が2倍になるなら、保守コストは半分になって初めて長期的な生産性が維持される。
- コード生成量が3倍になるなら、保守コストは3分の1になる必要がある。

以下の観点を常に意識して実装を行うこと：

1. 現在の開発生産性の定義
   - チームが「生産性」と呼んでいるものは何か
   - それはコード生成速度なのか、価値提供速度なのか、保守後の持続的速度なのか
   - その定義に含まれていないコストは何か

2. AI導入による短期効果
   - AIによって増えるアウトプットは何か
   - コード量、PR数、機能数、レビュー量、テスト量、設計判断量はどう変化するか
   - 短期的な速度向上が本当に価値提供につながる条件は何か

3. 保守コストへの影響
   以下の観点で、AI生成コードが保守コストを増やすか減らすかを評価すること。
   - コードの理解しやすさ
   - 設計の一貫性
   - テスト容易性
   - 変更容易性
   - デバッグ容易性
   - 依存関係の複雑さ
   - レビュー負荷
   - ドメイン知識との整合性
   - 既存アーキテクチャとの整合性
   - 将来の開発者が読んだときの認知負荷

4. 生産性モデル
   以下の3つのケースを比較検討すること。
   - ケースA：AIがコード生成量を増やすが、保守コストも増える
   - ケースB：AIがコード生成量を増やすが、保守コストは従来と同じ
   - ケースC：AIがコード生成量を増やし、保守コストを逆比例で下げる

   それぞれについて、短期的なメリット、中期的なリスク、長期的な生産性、AIを途中でやめた場合に残る負債を評価すること。

5. 導入可否の判断基準
   - コード作成速度だけでなく、保守コストを測定できているか
   - AI生成コードのレビュー品質を維持できるか
   - AIが設計品質を改善しているか
   - AIがテスト、リファクタリング、理解支援にも使われているか
   - AIなしでも保守可能なコードが残るか
   - ツールをやめた後も負債だけが残る状態になっていないか

6. 必要なメトリクス
   変更リードタイム、バグ修正時間、PRレビュー時間、差し戻し率、リファクタリング頻度、テスト失敗率、変更時の影響範囲、新規開発と保守作業の時間比率、コード理解にかかる時間、AI生成コード起因の障害率などを考慮すること。

重要：
コード生成速度の向上を、それ単独で成功と判断しないこと。
長期的な生産性は、生成されたコードを将来どれだけ安く、安全に、素早く変更できるかによって決まる。
AIの価値は「より多くのコードを書くこと」ではなく、「将来の保守コストを下げながら価値提供速度を上げること」にある。
