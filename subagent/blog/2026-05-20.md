# Parallel Proposal Generation: LLM Agent における並列変更提案と機械的適用の分離

## Abstract

LLM agent が複数の subagent を並列実行する際、共有 worktree に対する同時書き込みは semantic conflict を引き起こす。本稿では、subagent を「変更トランザクションの提案者」として扱い、worktree への反映を親 agent の apply engine に直列化する設計（Parallel Proposal Generation）を提案する。

主な貢献は次の4点である。

1. **トークン消費の分離.** subagent が生成した diff を ResultStore に保存し、親 LLM には summary のみを返すことで、並列 subagent 数の増加によるトークン消費を O(diff_size × N) から O(summary_size × N) に削減する。
2. **Stale proposal の機械的検出.** Base hash check により、並列提案生成後に前提が古くなった proposal を検出・reject する。
3. **軽量 semantic conflict 検出.** subagent が申告する semantic reads/writes と apply log を比較し、意味的衝突を部分的に検出する。完全な依存解析ではなく、実用的な検出率と実装コストのバランスを狙う。
4. **Authority による scope 制約の強制.** Prompt 指示ではなく tool の提供・非提供により権限を保証し、`write_scope` と patch 本文から抽出した actual touched paths を照合する。

各設計判断（design decision）には妥当性の根拠を示し、未解決課題を Discussion として整理する。

## 1. Problem: 共有 worktree における並列書き込み

LLM agent に subagent を持たせる動機は明確である。複数の観点からコードを調査し、別々の仮説を検証し、修正方針を比較できる。単一 agent の逐次処理に比べ、探索空間のカバー率は大幅に向上する。

しかし、subagent にファイル編集を許可すると、共有 worktree に対する並列 write が発生する。問題は Git conflict にとどまらない。ある subagent が API の戻り値を変更し、別の subagent が古い API 前提で別ファイルを修正する場合、Git 上は conflict しなくても意味的には壊れる。この種の semantic conflict はファイル単位の排他制御では防げない。

さらに、subagent が生成した diff をそのまま親 LLM の context に流すと、diff のサイズに比例してトークン消費が増大し、並列性の恩恵が相殺される。

## 2. Approach: Parallel Proposal Generation

```text
subagent A ── 調査 ── patch proposal ──┐
subagent B ── 調査 ── no_change     ──┤── ResultStore ── ApplyQueue ── worktree
subagent C ── 調査 ── needs_decision ──┘
```

基本方針は次の3点に集約される。

1. **subagent は提案者であり、編集者ではない。** subagent は並列に調査・分析し、必要なら structured な変更案（patch proposal）を返す。ただし worktree への反映は行わない。
2. **親 agent は diff を読まない。** patch 本文は ResultStore に保存され、親には `result_id` と summary だけが返る。
3. **適用は直列かつ機械的。** ApplyQueue が pending result を schema validation、authority check、base hash check、path safety、semantic conflict check、`git apply --check` を経て適用する。

以下、各構成要素を設計上の判断（design decision）とともに述べる。

## 3. SubagentResultV1: 出力の構造化

subagent の最終出力は自由文ではなく、structured result として扱う。outcome は次の5種類。

| outcome | 意味 |
|---------|------|
| `no_change` | 調査の結果、変更不要と判断 |
| `patch` | 変更案を含む patch proposal |
| `blocked` | 権限または情報不足で進行不可 |
| `needs_decision` | 設計判断が必要 |
| `observation` | 調査結果のみ（修正なし） |

この分類の設計意図は、subagent に「必ず patch を返す」という圧力をかけないことにある。変更不要という結論も有用な結果であり、`no_change` として明示できる。

`patch` outcome は diff 本文に加えて、次の metadata を持つ。

```text
base file hashes     -- proposal 生成時のファイル状態
touched paths        -- 変更対象パス
semantic reads       -- 読み取り前提とした意味的要素
semantic writes      -- 変更する意味的要素
semantic assumptions -- 暗黙の前提
semantic effects     -- 予想される影響
public surface delta -- 公開 API/型の変更
validation commands  -- 適用後の検証コマンド
risk level           -- リスク評価 (low/medium/high)
```

この metadata により、apply engine は diff を LLM に解釈させることなく、機械的な検証判定が可能になる。

**Design Decision:** なぜ semantic metadata を subagent に申告させるのか。理想は static analysis による自動抽出だが、LLM が持つ文脈理解を活用できる点で、自己申告＋事後検証の組み合わせが実用的である。subagent は「何を読み、何を書き、何を前提としたか」を表明し、apply engine はその申告を検証する。

## 4. ResultStore: トークン消費の分離

patch 本文を親 LLM の context に流すと、subagent 数 × diff サイズのトークン消費が発生する。ResultStore はこれを分離する。

subagent が返す `patch.body` は `.pi/subagent-results/` 配下に保存され、内部的に `patch.ref` が割り当てられる。subagent が直接 `patch.ref` を指定することは禁止している（任意パスの指し示しリスクを排除）。

親 agent の mailbox に返るのは summary のみ。

```json
{
  "kind": "subagent_result_available",
  "result_id": "sar_...",
  "outcome": "patch",
  "summary": "expired token handling fix",
  "touched_paths": ["src/auth/token.ts"],
  "semantic_risk": "low"
}
```

**Design Decision:** この設計により、subagent が大きな diff を生成しても親 agent の context を圧迫しない。並列 subagent 数の増加が親のトークン予算に与える影響を、O(diff_size × N) から O(summary_size × N) に削減できる。

ResultStore に保存された JSON は apply 時に再検証される（result_id の形式、filename との一致、status の enum 妥当性、patch.ref の配置先）。保存済み result を過度に信用しない姿勢は、apply pipeline 全体の信頼性に寄与する。

## 5. Authority: 権限の実装による保証

subagent の authority は3つの mode で制御する。

| mode | 許可操作 |
|------|----------|
| `read_only` | 読み取りのみ |
| `propose_patch` | 読み取り + patch proposal |
| `edit` | 読み取り + patch proposal + 直接編集 |

`read_only` / `propose_patch` では、tool が allowlist で制限される。書き込み系 tool や shell 実行系 tool は提供されない。

**Design Decision:** 「ファイルを編集しないでください」という prompt 指示だけでは権限を保証できない。LLM は利用可能な tool を使う傾向があり、書き込み可能な tool が渡されていれば意図せず編集する可能性がある。権限の保証は実装レベル（tool の提供・非提供）で行う必要がある。

外部 Pi（external Pi）のように authority enforcement が弱い実行形態では `authority_enforced: false` とし、その result は自動 apply の対象としない。これは信頼の程度に応じた段階的な安全性確保である。

`write_scope` により「どのパス範囲への変更提案を許可するか」を制約し、`semantic_scope` により「どの意味的領域に介入してよいか」を制約する。これらは apply 時に patch 本文から抽出した actual touched paths と照合される。

## 6. ApplyQueue: 直列かつ機械的な適用

ApplyQueue は ResultStore の pending result を次の pipeline で処理する。

```text
recovery ── status確認 ── outcome処理 ── markApplying
  └─ patch.ref検証 ── size check ── path抽出と照合
     └─ write_scope check ── semantic_scope check
        └─ base hash check ── public surface check
           └─ authority check ── semantic conflict check
              └─ validation allowlist check
                 └─ git apply --check ── git apply
                    └─ validation実行 ── markApplied ── semantic log追記
```

自動 apply を止める条件（review へ回す条件）：

```text
write_scope が未指定
authority_enforced が false (external Pi 由来)
risk level が high
breaking / unknown public surface change
execution-sensitive path (.husky/** 等)
workspace_cwd mismatch
semantic conflict 検出
apply engine 内部例外
```

**Design Decision:** すべての変更を人間または親 LLM が review するのは非効率である。low risk・scope 内・base hash 一致・validation 通過の patch は自動 apply の候補とし、review 対象を高リスクに絞る。この分類により、「安全性を保ちながら自動化の恩恵を得る」という実用的な妥協点を設定している。

### 6.1 Path Safety

subagent の自己申告（`scope.touched_paths`）を信用せず、patch 本文から actual touched paths を抽出する。

```text
reject 対象:
  ../outside.ts, /absolute/path.ts, C:\outside.ts
  .git/** (全面禁止), .pi/** (全面禁止)
  .husky/** (review へ回す)
```

### 6.2 Base Hash Check

並列 proposal では、subagent が古いファイル状態を前提に patch を作る可能性がある。patch に base file hash を含め、apply 時に現在のファイル hash と比較する。不一致の場合は `base_hash_mismatch` として reject する。

この仕組みは並列 subagent と相性が良い。同時に調査を開始しても、apply 時点で前提が古くなっていれば機械的に検出できる。

### 6.3 Validation Command

subagent が任意の command を返し、親が実行するのは危険である。`authority.allowed_commands` に含まれる command（`npm_script`, `shell_allowlisted`）のみ実行する。

validation 失敗時は `git apply -R` で rollback を試みる（best-effort）。validation 後の worktree 放置を防ぐための重要な措置である。

## 7. Semantic Conflict Detection

semantic conflict の完全な消除は目標ではない。検出できる範囲を metadata 化し、危険な proposal を review / retry に回す。

apply 済みの patch は semantic apply log に記録され、後続 patch と次のルールで比較される。

```text
applied が X を write, incoming が X を read  → require_regeneration
applied が X を write, incoming も X を write  → require_review
public surface が breaking / unknown           → require_review
risk が high                                   → require_review
```

**Design Decision:** 完全な依存関係解析（依存グラフの構築など）はコストが高い。semantic metadata の宣言と比較による軽量検出は、妥当な実用性と実装コストのバランスである。検出できないから安全、とは扱わない。

### 7.1 Public Surface Delta

public surface（TypeScript export, REST API, GraphQL schema, DB schema, config schema, CLI, event payload, file format）の変更は特別扱いする。patch 本文から簡易的に delta を検出し、subagent の宣言と比較する。未宣言の public surface change は reject する。

この detector は安全性の証明ではなく gate である。検出できない変更が安全であるとは仮定しない。

## 8. State Machine

result の状態は agent の状態とは独立に管理する。

```text
pending → applying → applied
                  └→ needs_review
         └→ rejected
         └→ superseded
```

`markApplying()` による二重適用防止、`recoverStaleApplying()` によるクラッシュリカバリ、`applyOne()` の例外境界による `apply_engine_exception` 処理は、pipeline の堅牢性を支える基盤である。

## 9. Retry: 提案の再生成

base hash mismatch や semantic conflict で reject された場合、proposal の再生成が必要になる。しかし patch proposal を返した subagent は多くの場合 completed 状態である。

`retry_agent_result` は次のように動作する。

```text
元 agent が open + non-terminal → followup 送信
元 agent が terminal            → 新規 retry subagent を spawn
```

retry subagent は reject reason と invalidated context を受け取り、新しい proposal を生成する。これにより、親 LLM が diff を読んで修正するのではなく、機械的な再提案ループを構成できる。

## 10. Discussion

### 10.1 Semantic Metadata の信頼性

現在の reads/writes/effects/assumptions は subagent の自己申告である。LLM の申告をどの程度信用できるかは未解決の問題である。

具体的な懸念は二つある。第一に、subagent が自身の変更の影響範囲を過小評価する可能性である。LLM は局所的な変更の副作用を見落としやすく、semantic writes に含まれない暗黙の依存が存在し得る。第二に、悪意はないが不完全な申告により、apply engine が誤って安全と判断する false negative のリスクである。

**改善方向:** semantic fingerprint の導入が考えられる。これは repository の現在状態から自動算出される署名であり、subagent の申告と照合する。たとえば、subagent が「X を read した」と申告した場合、X の現在の hash を記録し、apply engine がその妥当性を検証する。申告と実際の差分があれば、信頼性の指標として使える。

### 10.2 Public Surface Detection の限界

現在の detector は patch 本文の pattern matching に基づく gate である。TypeScript の `export`、`package.json` の `version`、OpenAPI の `paths` などを regex で検出する。

この方法には明確な限界がある。
- 追加 export を含むリファクタリングを見逃す可能性
- barrel file（index.ts）経由の再 export の追跡が困難
- 型の互換性変更（narrowing/widening）の検出不可
- runtime behavior の変更（side effect の有無など）の検出不可

**改善方向:** domain-specific checker の追加。TypeScript AST による export 解析、OpenAPI diff tool（oasdiff 等）の統合、GraphQL schema の breaking change 検出。ただし、各 checker は検出率の向上に過ぎず、安全性の証明ではない。この認識を維持することが、過信を防ぐ鍵である。

### 10.3 External Pi と Worktree Isolation

外部 Pi（external Pi）では authority enforcement が弱い。`authority_enforced: false` の場合、subagent はツール制限を受けず、同じ worktree を直接編集する可能性が残る。現在は「その result は自動 apply しない」という消極的防衛に留まっている。

これは現在の設計における最大の未解決課題である。解決には二つの方向がある。

1. **Spawn 前後の worktree snapshot.** subagent 起動前の worktree 状態を記録し、完了後に diff を取る。予期しない変更があれば検出できる。ただし、実行中の変更は検出できない。

2. **Isolated worktree (git worktree).** subagent ごとに独立した git worktree を割り当てる。物理的に分離されるため、並列書き込みの危険が消除される。ただし、worktree の作成・破棄コストと、親 worktree との同期コストが発生する。

### 10.4 Risk Level と自動適用の境界

現在の設計では、low risk のみ自動 apply の候補としている。しかし「low」の定義はどこにあるか。

実際の運用では、次の条件をすべて満たすものを「low risk」としている。
- `write_scope` が明示的に指定されている
- `authority_enforced` が true
- base hash が一致
- public surface delta が none または backward-compatible
- validation command が成功
- semantic conflict が検出されない

これらを満たさない場合の対応は、`needs_review`（人間または親 LLM の判断を待つ）と `require_regeneration`（subagent に再生成させる）に分かれる。

**議論の余地:** medium risk まで自動 apply を許すかどうか。判断基準として、「自動 apply 後の rollback 成功率」と「review 率の低下による効率化」のトレードオフを定量的に評価する必要がある。

### 10.5 LLM と機械的検証の役割分離

この設計の理論的基盤は、LLM の出力が本質的に非決定的であるという前提にある。同じ prompt でも異なる出力を生成する可能性があり、この非決定性は LLM の柔軟性の源泉でもあり、安全性の課題でもある。

```text
LLM の領域（非決定的）:
  調査、仮説生成、変更案生成、semantic metadata の申告

機械の領域（決定的）:
  schema validation、authority enforcement、path safety、
  base hash check、git apply、validation 実行
```

この分離により、LLM の非決定性を探索に使い、mutation の正確性は決定的な処理に保証させる。非決定性を「消す」のではなく、影響を探索フェーズに限定する設計である。

この分離が崩れる場面は、apply engine が例外を投げた時である。`apply_engine_exception_after_patch_applied` のように、patch 適用後に例外が発生した場合、rollback は best-effort となり、機械的保証の境界が曖昧になる。この境界の管理は今後の課題である。

## 11. Summary

```text
parallel proposal generation ── 並列性は探索・提案に使う
serial mechanical application ── mutation は直列・機械的
semantic-aware conflict control ── 意味的衝突は軽量 metadata で検出
```

今後の拡張方向：
- **semantic fingerprint**: 申告の妥当性を repository 状態と照合
- **domain-specific public surface detection**: AST/API schema 解析による検出精度向上
- **external worktree isolation**: git worktree を用いた物理的分離
