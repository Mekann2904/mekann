# Parallel Proposal Generation: Subagent の並列探索と安全な機械的適用

LLM agent に subagent を持たせると、調査や分析を並列化できます。複数の観点からコードを読ませたり、別々の仮説を検証させたり、修正方針を比較させたりできるため、単一 agent が順番に作業するよりも探索効率は上がります。

一方で、subagent にそのままファイル編集まで任せると、共有 worktree に対する並列 write が発生します。これは単純な Git conflict だけでなく、より見えにくい semantic conflict を引き起こします。

たとえば、ある subagent が API の戻り値を変更し、別の subagent が古い API の前提で別ファイルを修正する場合、Git 上は conflict しないかもしれません。しかし意味的には壊れています。この種の衝突は、ファイル単位の排他制御だけでは防げません。

現在の実装では、この問題に対して次の方針を採っています。

**subagent は並列に調査し、必要なら structured な変更案を返す。ただし共有 worktree への反映は親側の apply engine が直列に行う。**

つまり、subagent を「編集者」ではなく、**変更トランザクションの提案者**として扱います。

---

# 全体像

現在の設計は、次のような構成です。

```text
subagent A ── 調査 ── patch proposal
subagent B ── 調査 ── no_change
subagent C ── 調査 ── needs_decision

ResultStore
  └─ structured result と patch body を保存

ApplyQueue
  ├─ schema validation
  ├─ authority check
  ├─ path safety check
  ├─ base hash check
  ├─ semantic conflict check
  ├─ git apply --check
  ├─ git apply
  └─ validation command execution
```

重要なのは、親 agent が diff 本文を読まないことです。

subagent は変更案を返しますが、その内容は会話にそのまま流しません。patch 本文は ResultStore に保存され、親 agent には `result_id` と summary だけが返ります。

これにより、親 LLM のトークン消費を抑えながら、subagent の並列性を利用できます。

---

# Structured Result

subagent の最終出力は自由文ではなく、`SubagentResultV1` という structured result として扱われます。

現在の outcome は次の5種類です。

```text
no_change
patch
blocked
needs_decision
observation
```

それぞれの意味は次の通りです。

`no_change` は、調査した結果、変更不要だった場合です。
`patch` は、変更案がある場合です。
`blocked` は、権限や情報が不足していて進められない場合です。
`needs_decision` は、設計判断が必要な場合です。
`observation` は、修正ではなく調査結果だけを返す場合です。

この分類により、subagent が無理に patch を返す必要がなくなります。調べた結果、変更しない方がよい場合は `no_change` として明示できます。

`patch` outcome では、単なる diff だけでなく、次のような metadata を持ちます。

```text
base file hashes
touched paths
semantic reads
semantic writes
semantic assumptions
semantic effects
public surface delta
validation commands
risk level
```

この metadata があることで、apply engine は diff の意味を LLM に読ませずに、機械的に検証できます。

---

# ResultStore

patch 本文を会話に流すと、親 LLM が大量のトークンを消費します。
そこで現在の実装では、structured result を `SubagentResultStore` に保存します。

patch proposal の場合、subagent は `patch.body` を返します。
ResultStore はその body を `.pi/subagent-results/` 配下の patch file として保存し、内部的に `patch.ref` を割り当てます。

subagent が直接 `patch.ref` を指定することは許可していません。
これは、subagent が任意のファイルパスを指してしまうリスクを避けるためです。

mailbox に返すのは、次のような小さな summary だけです。

```json
{
  "kind": "subagent_result_available",
  "result_id": "sar_...",
  "agent_path": "/root/auth/fix",
  "outcome": "patch",
  "summary": "expired token handling fix",
  "touched_paths": ["src/auth/token.ts"],
  "semantic_risk": "low"
}
```

親 agent はこの `result_id` を使って、必要なら `apply_agent_results` を呼びます。
diff 本文は通常の会話 context に入りません。

---

# Authority

現在の実装では、subagent に authority を持たせています。

主な mode は次の3つです。

```text
read_only
propose_patch
edit
```

`read_only` は調査のみです。
`propose_patch` は patch proposal を返せますが、実ファイルは直接編集しません。
`edit` は実編集を許可する mode ですが、共有 worktree では慎重に扱うべきものです。

`read_only` / `propose_patch` では、tool は allowlist で制限されます。
読み取り系の tool だけを許可し、write 系 tool や shell 実行系 tool を使わせません。

この制御は重要です。
「ファイルを編集しないでください」という prompt だけでは、権限の保証にはなりません。実装上、書き込み可能な tool が渡されていれば、LLM が意図せず編集してしまう可能性があります。

また、external Pi のように authority enforcement が弱い実行形態では、`authority_enforced: false` として扱います。そのような result は自動 apply せず、review に回します。

---

# ApplyQueue

`ApplyQueue` は、ResultStore に保存された pending result を処理します。

現在の apply flow は、おおむね次の通りです。

```text
1. stale applying result の recovery
2. result status の確認
3. workspace_cwd の確認
4. outcome の処理
5. markApplying
6. patch.ref の検証
7. patch size check
8. actual touched paths extraction
9. declared touched paths との照合
10. write_scope check
11. review-only path check
12. semantic_scope check
13. base hash check
14. public surface delta check
15. authority_enforced check
16. semantic conflict check
17. validation command allowlist check
18. git apply --check
19. git apply
20. validation 実行
21. markApplied
22. semantic log 追記
```

この処理により、親 agent は diff 本文を読まずに、pending result を機械的に処理できます。

もちろん、すべてを自動 apply するわけではありません。
危険な result は `needs_review` に落とします。

たとえば、次のような場合は自動 apply しません。

```text
write_scope が未指定
external Pi 由来で authority が enforcement されていない
high risk
breaking / unknown public surface change
.husky/** のような execution-sensitive path
workspace_cwd mismatch
semantic conflict
apply engine exception
```

---

# Path Safety

この設計で特に重要なのが path safety です。

subagent が返す `scope.touched_paths` は信用しません。
patch 本文から actual touched paths を抽出し、declared touched paths と一致するかを確認します。

現在は `extractTouchedPathsFromPatchStrict()` によって、patch 内の path を strict に検査しています。

次のような path は reject されます。

```text
../outside.ts
/absolute/path.ts
C:\outside.ts
.git/config
.pi/subagent-results/...
```

`.git/**` と `.pi/**` は全面的に禁止されています。
`.husky/**` のように実行に関わる path は、auto apply せず review に回します。

この仕組みにより、たとえ `write_scope` が広く設定されていても、Git metadata や extension の内部状態を守れます。

---

# Base Hash Check

並列に proposal を生成すると、subagent が古いファイル状態を前提に patch を作る可能性があります。

そこで patch proposal には base file hash を含めます。
apply 時には、actual touched paths に対して base hash が存在するかを確認し、現在のファイル hash と比較します。

```text
hash が一致する
  → apply 続行

hash が一致しない
  → base_hash_mismatch

hash が不足している
  → base_hash_mismatch

新規ファイル
  → /dev/null からの patch なら例外
```

これにより、古い proposal を機械的に reject できます。

これは並列 subagent にとって重要です。
複数の subagent が同時に調査を始めた場合でも、先に apply された変更によって後続 proposal が stale になったことを検出できます。

---

# Validation Command

patch を apply した後は validation を実行します。

ただし、subagent が任意の command を返し、それを親が実行するのは危険です。
そのため、現在の実装では `authority.allowed_commands` に含まれる command のみ実行します。

command は canonical key によって比較されます。

```text
npm_script
shell_allowlisted
```

`validation.required` も schema validation されます。
command が明示されている場合はそれを使い、ない場合は suggested command から convention によって解決を試みます。

validation が失敗した場合は、`git apply -R` による rollback を試みます。
rollback は best-effort ですが、validation failure 後に worktree を放置しないための重要な処理です。

---

# Semantic Conflict

この設計では、semantic conflict を完全に消すことは目指していません。
代わりに、検出できる範囲を metadata 化し、危険な proposal を review / retry に回します。

patch proposal には次の semantic metadata を持たせます。

```text
reads
writes
assumptions
effects
public_surface_delta
risk
```

apply 済みの patch は semantic apply log に記録されます。

後続 patch に対しては、semantic log と incoming patch の metadata を比較します。

代表的なルールは次の通りです。

```text
適用済み patch が X を write
incoming patch が X を read
  → require_regeneration

適用済み patch が X を write
incoming patch も X を write
  → require_review

public surface が breaking / unknown
  → require_review

risk が high
  → require_review
```

この仕組みにより、親 LLM に毎回 diff を読ませるのではなく、semantic metadata の比較で危険な proposal を止められます。

---

# Public Surface Delta

semantic conflict の中でも、public surface の変更は特別扱いします。

対象はたとえば次のようなものです。

```text
TypeScript export
REST API
GraphQL schema
database schema
config schema
CLI
event payload
file format
```

現在の実装では、patch 本文から簡易的に public surface delta を検出します。
TypeScript の export、schema、routes、openapi、package.json、tsconfig などが対象です。

subagent が宣言した `public_surface_delta` と、patch 本文から検出した actual delta を比較します。
未宣言の public surface change が見つかった場合は reject します。

この detector は「安全性の証明」ではありません。
あくまで、危険そうな変更を検出する gate です。
検出できないから安全、という扱いにはしていません。

---

# ResultStore の再検証

ResultStore に保存された JSON も、apply 時には再検証します。

現在の実装では、少なくとも次を確認します。

```text
result_id が安全な形式である
filename と JSON 内 result_id が一致する
status が known enum である
stored result が SubagentResultV1 として valid である
patch.ref が ResultStore 配下にある
workspace_cwd が現在の cwd と一致する
```

ResultStore 内の壊れた JSON は list 時に skip されます。
直接 load する場合は error として扱われます。

これにより、保存済み result を apply queue が過度に信用しないようにしています。

---

# 状態遷移

result の状態は agent の状態とは別に管理します。

主な status は次の通りです。

```text
pending
applying
applied
rejected
needs_review
superseded
```

apply 開始時には `markApplying()` を呼びます。
これにより、同じ result が二重に apply されることを防ぎます。

もし `pending -> applying` の直後にプロセスが落ちると、result が `applying` のまま残る可能性があります。
そのため、`recoverStaleApplying()` が古い `applying` result を `needs_review` に回します。

また、`applyOne()` 全体は例外境界で守られており、予期しない例外は `apply_engine_exception` として review に落ちます。
patch 適用後に例外が起きた場合は、`apply_engine_exception_after_patch_applied` として扱い、可能なら rollback を試みます。

---

# Retry

base hash mismatch や semantic conflict が起きた場合、proposal を再生成したいことがあります。

ただし、patch proposal を返した subagent は多くの場合 completed になっています。
completed agent に followup できない場合、retry が詰まります。

現在の実装では、`retry_agent_result` が次のように動きます。

```text
元 agent が open かつ non-terminal
  → followup を送る

元 agent が terminal
  → 新しい retry subagent を spawn する
```

retry subagent は、元 result の reject reason や invalidated context を受け取って、新しい proposal を生成します。

これにより、親 agent が diff を読んで修正するのではなく、再 proposal として処理できます。

---

# この方法で得られる恩恵

この設計の大きな恩恵は、**並列性と安全性を分離できること**です。

subagent の並列性は、探索・調査・proposal generation に使います。
一方で、worktree に対する mutation は、親側の apply engine に集約します。

これにより、次の利点が得られます。

## 1. 親 LLM のトークン消費を抑えられる

patch 本文を親 LLM に読ませません。
ResultStore に保存し、親には `result_id` と summary だけ返します。

そのため、subagent が大きな diff を生成しても、親 agent の context を圧迫しません。

## 2. 並列探索ができる

subagent は同時に複数走らせられます。
それぞれが別の仮説、別の対象領域、別の修正方針を調べられます。

ただし、実際の write は apply queue が直列に行うため、共有 worktree の race condition は避けられます。

## 3. stale proposal を機械的に reject できる

base hash check により、古いファイル状態に基づいた proposal を検出できます。

これは並列化と相性が良いです。
subagent が同時に調査を始めても、apply 時点で前提が古くなっていれば reject できます。

## 4. path scope を強制できる

subagent の自己申告ではなく、patch 本文から actual touched paths を抽出します。
それを authority の `write_scope` と照合します。

これにより、「この subagent はこの範囲だけ変更提案してよい」という制約を実装できます。

## 5. semantic conflict を review / retry に回せる

semantic reads/writes と semantic apply log により、意味的な依存関係の一部を検出できます。

完全ではありませんが、少なくとも次のようなケースは機械的に止められます。

```text
A が X を変更した
B が古い X を前提にしている
```

この場合、B をそのまま apply せず、retry に回せます。

## 6. 危険な変更だけ review に回せる

すべてを人間や親 LLM が見る必要はありません。

low risk で scope 内、base hash も一致し、validation も通る patch は自動 apply の候補になります。
一方で、public surface change、high risk、external Pi 由来、execution-sensitive path などは review に回します。

つまり、review の対象を絞れます。

## 7. 「LLM の判断」と「機械的検証」を分離できる

LLM は探索や提案に向いています。
一方で、path check、hash check、schema validation、command allowlist、`git apply --check` などは機械的に行うべきです。

この設計では、それぞれの役割が分かれています。

```text
LLM:
  調査
  仮説生成
  変更案生成
  semantic metadata の提案

Apply engine:
  schema validation
  authority enforcement
  path safety
  base hash check
  git apply
  validation execution
```

この分離により、LLM の柔軟性を使いつつ、実際の mutation は deterministic な処理に寄せられます。

# 議論したいポイント

この方式には明確な利点がありますが、まだ議論すべき点もあります。

まず、semantic metadata をどこまで信用するかです。
現在の reads/writes/effects/assumptions は、基本的には subagent の自己申告です。今後は semantic fingerprint を導入し、現在の repository 状態と照合できるようにする必要があります。

次に、public surface detector の精度です。
現在は gate としては有用ですが、完全な安全性の証明ではありません。TypeScript AST、OpenAPI diff、GraphQL schema diff、DB migration check など、domain-specific checker を増やす余地があります。

また、external Pi の扱いも議論点です。
現在は authority enforcement が弱いため自動 apply しません。しかし、同じ worktree を直接編集してしまう可能性は残ります。これを防ぐには、spawn 前後の worktree snapshot や isolated worktree が必要になります。

最後に、どの risk level まで auto apply するかも運用上の判断です。
low risk だけに限定するのか、validation が十分なら medium risk も許すのか。ここはプロジェクトの許容リスクによって変わります。

# まとめ

現在の実装は、subagent を直接編集者として扱うのではなく、**structured な変更提案者**として扱います。

subagent は並列に調査し、必要なら patch proposal を返します。
親 agent は diff を読まず、ResultStore と ApplyQueue を通じて機械的に処理します。
安全なものは apply し、危険なものは review や retry に回します。

この方式の本質は次です。

```text
parallel proposal generation
serial mechanical application
semantic-aware conflict control
```

LLM に並列で考えさせる。
しかし、共有 worktree への mutation は deterministic な apply engine に閉じ込める。

この分離によって、subagent の並列性を活かしながら、直接並列編集の危険を避けられます。
低リスクで scope が明確な patch については自動 apply の候補にでき、高リスクな変更は review に回せます。

今後の主な拡張余地は、semantic fingerprint、domain-specific public surface detection、external worktree isolation です。
これらが入ると、より広い種類の変更に対して、安全に parallel proposal generation を使えるようになります。
