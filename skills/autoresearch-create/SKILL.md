# Autoresearch

自律的実験ループ: アイデアを試し、良ければ採用し、悪ければ破棄し、停止されるまで続ける。

## ツール

- **`autoresearch_init`** — セッションの初期化（名前、指標、単位、方向）。最適化対象が変わったら再初期化する。
- **`autoresearch_run`** — コマンドを実行し、実行時間と出力を記録する。`autoresearch.checks.sh` が存在すれば成功後に自動実行する。
- **`autoresearch_log`** — 結果を記録する。`keep` は自動で git commit する。`discard` / `crash` / `checks_failed` は自動でコード変更を revert する（autoresearch 関連ファイルは保護される）。

## セットアップ手順

1. **目的・指標・コマンド・対象ファイル・制約** をユーザーに確認する（または推測する）。
2. `git checkout -b autoresearch/<goal>-<date>` でブランチを作成する。
3. ソースファイルを読み、ワークロードを深く理解してから書き込みを始める。
4. `autoresearch.md` と `autoresearch.sh` を作成する（下記参照）。両方を commit する。
5. `autoresearch_init` → ベースラインを `autoresearch_run` → `autoresearch_log` → **直ちにループ開始**。

### `autoresearch.md`

セッションの中核。コンテキストを持たない新規エージェントがこのファイルを読むだけでループを効果的に実行できるように、時間をかけて充実させること。

```markdown
# Autoresearch: <目的>

## 目的
<何を最適化するのか、ワークロードの具体的な説明。>

## 指標
- **主指標**: <名前> (<単位>, 低い/高い方が良い) — 最適化対象
- **副指標**: <名前>, <名前>, ... — 独立したトレードオフ監視

## 実行方法
`./autoresearch.sh` — `METRIC name=数値` 行を出力する。

## 対象ファイル
<エージェントが変更する可能性のある全ファイルと、それぞれの役割。>

## 禁止事項
<触れてはいけないもの。>

## 制約
<ハードルール: テストは必ず通す、新しい依存関係は追加しない、など。>

## 試したこと
<実験が蓄積するたびにこのセクションを更新する。主な成功、失敗、
アーキテクチャ上の知見を記録し、エージェントが同じ失敗を繰り返さないようにする。>
```

`autoresearch.md` は定期的に更新すること。特に「試したこと」セクションは、再開エージェントが完全なコンテキストを持てるように保つ。

### `autoresearch.sh`

Bash スクリプト（`set -euo pipefail`）。素早い事前チェック（1秒以内で構文エラーを検出）、ベンチマークを実行、構造化行を stdout に出力する。スクリプトは高速に保つこと — 毎秒が数百回の実行で乗算される。

**高速でノイズ多いベンチマーク**（5秒未満）の場合、スクリプト内でワークロードを複数回実行し中央値を報告する。これにより安定したデータポイントが得られる。遅いワークロード（ML学習、大規模ビルド）では単回実行でよい。

#### 構造化出力

- `METRIC name=value` — 主指標（`autoresearch_init` の `metric_name` と一致させること）および副指標。`autoresearch_run` が自動的にパースする。

#### 最適化に役立つスクリプト設計

スクリプトは **次のイテレーションでより良い判断をするのに役立つデータ** を出力するべきだ。各実行後にどこに注目すべきか知るために必要なものを考えること:

- ワークロードに明確な段階がある場合: フェーズごとの所要時間
- チェックが異なる形で失敗する可能性がある場合: エラー数、失敗カテゴリ、テスト名
- 関連する場合: メモリ使用量、キャッシュヒット率、その他のランタイム診断
- ボトルネックやリグレッションの特定に役立つドメイン固有の情報

スクリプトは毎回同じコードを実行するが、より多くの信号が必要になった場合は **ループ中に更新** できる。学ぶにつれて計測を追加する。

#### 失敗の記録

`discard` / `crash` / `checks_failed` の実行は revert される — コード変更は消える。`autoresearch.jsonl` に残る記録は `description` と `memo` だけである。何を試し、なぜ失敗したかを記録しないと、将来のイテレーションが同じ失敗を再発見する時間を無駄にする。

**失敗とクラッシュは厚く記録すること。** `description` に何を試したかを書き、`memo` に学んだことを書き、`autoresearch.md` の「試したこと」セクションも更新する。

### `autoresearch.checks.sh`（任意）

正確性チェック用の Bash スクリプト（`set -euo pipefail`）。テスト、型チェック、lint など。**ユーザーの制約が正確性検証を要求する場合のみ** 作成する（例: 「テストは必ず通す」「型チェック必須」）。

このファイルが存在する場合:
- ベンチマーク成功後、`autoresearch_run` が自動的に実行する。
- チェックが失敗した場合、`autoresearch_run` が明確に報告する — `checks_failed` で記録すること。
- チェックの実行時間は主指標に **影響しない**。
- チェックが失敗した場合、`keep` は選択できない。
- 個別のタイムアウトがある（デフォルト 300秒、`checks_timeout_seconds` で変更可能）。

**出力は最小限に。** 失敗時にエージェントに返されるのは末尾80行のみ。冗長な成功出力は抑制し、エラーだけを通すこと。

```bash
#!/bin/bash
set -euo pipefail
# 例: テストと型チェックを実行 — 成功出力は抑制、エラーのみ表示
pnpm test --run --reporter=dot 2>&1 | tail -50
pnpm typecheck 2>&1 | grep -i error || true
```

## ループ規則

**永遠にループし続ける。** 「続けますか？」と確認しない — ユーザーは自律的な作業を期待している。

- **主指標が最優先。** 改善された → `keep`。悪化・同等 → `discard`。副指標はこれに影響しないことが多い。
- **すべての実行を `description` と `memo` で記録。** 何をしたかではなく、何を学んだかを記録する。次のイテレーションや、新規エージェントがセッションを再開する際に役立つ情報を残す。
- **シンプルが良い。** 同等の性能ならコードを削除した方が `keep`。わずかな利益のために醜い複雑さを追加するなら `discard` を検討する。
- **空回りしない。** 同じアイデアで繰り返し revert している？ 構造的に異なるアプローチを試す。
- **クラッシュ:** 自明な修正なら直す、そうでなければ記録して次へ進む。過剰な投資をしない。
- **行き詰まったら深く考える。** ソースファイルを読み直し、プロファイリングデータを研究し、CPUが実際に何をしているか推論する。最良のアイデアは深い理解から生まれ、ランダムなバリエーションの試行からは生まれない。
- **再開時:** `autoresearch.md` が存在する場合、それを読み + git log を確認し、ループを継続する。

**絶対に止まらない。** ユーザーは何時間も離れているかもしれない。中断されるまで続ける。

## アイデアバックログ

すぐには取り組まないが有望な最適化を見つけた場合、**`autoresearch.ideas.md` に箇条書きで追記** する。良いアイデアを失わないこと。

再開時（コンテキスト上限、クラッシュ）に `autoresearch.ideas.md` を確認する — 古い・試済みのエントリを刈り込み、残りを実験する。すべての道が尽きたら、ファイルを削除して最終サマリーを書く。

## 実験中のユーザーメッセージ

ユーザーが実験中にメッセージを送った場合、現在の `autoresearch_run` + `autoresearch_log` サイクルを完了してから、次のイテレーションでフィードバックを反映する。実行中の実験を放棄しない。
